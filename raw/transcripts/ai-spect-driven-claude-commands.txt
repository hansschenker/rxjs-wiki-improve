Command files are the slash commands - they appear in the / menu and inject a prompt when you invoke them. That's their only job.

What ~/ . claude/skills/ adds on top

1. Single source of truth
The command file is a thin wrapper (~2 lines). All the real logic lives in SKILL.md. If you want to update how /architect behaves, you edit one file (skills/architect/SKILL.md), not the command file.

2. Claude reads the SKILL.md automatically at session start
Because the command files reference the skill paths, Claude loads all 8 role definitions into context when the session starts. Claude "knows" all 8 roles before you type anything.

3. Proactive suggestion (the YAML description field)
The frontmatter description tells Claude when to suggest the skill unprompted:
description: This skill should be used when the user wants to gather requirements ...
or says "gather requirements", or "/analyst".
If you say "let's gather requirements" without typing /analyst, Claude can recognise the intent and invoke the skill via the Skill tool automatically.

4. Reuse from other skills or agents
A skill can reference another skill. An orchestrator agent can read SKILL.md files and delegate to the right role without needing the / menu at all.

do you agree that working with these SDLC roles is one of the best ways to work with Claude Cli

Yes, with some important caveats. Here's my honest assessment:

Why it genuinely works well

Artifacts solve the context window problem
Each role produces a file that persists. The developer doesn't need to remember the analyst conversation - it reads REQUIREMENT.md. If a session ends, the next one picks up from the artifact, not from memory.

Roles prevent scope creep
Claude without constraints tends to do everything at once - design while coding, review while building, speculate while specifying. Roles enforce a mandate: the developer implements the spec, does not redesign it.
This is how good human teams work too.

The chain grounds Claude in reality
REQUIREMENT.md constrains SPEC.md. SPEC.md constrains the code. CODE_REVIEW.md checks against both. Each stage reduces the surface area for hallucination and drift.

It maps to how good software is actually built
Requirements > design > implementation > review is not a Claude invention - it's how software engineering works. These roles just make the process explicit.

Where it has limits

Overkill for small tasks
A one-file script, a bug fix, a quick experiment - running /analyst through /writer adds friction that hurts more than it helps. The pipeline earns its overhead at project scale.

Artifacts go stale
If requirements change mid-project, SPEC.md and the code diverge. The chain only works if you maintain it. That discipline requires the human, not Claude.

The interview is only as good as the answers
REQUIREMENT.md captures what you said, not necessarily what you meant. Vague inputs from the analyst session produce vague outputs that compound through every downstream role.

It is not exploration
For understanding an unfamiliar codebase, debugging, or rapid prototyping, a free-form conversation with Claude is faster. The SDLC approach works best when you know what you're building.

The honest verdict

It is one of the best ways to use Claude CLI for intentional software development - projects where you care about architecture, maintainability, and traceability. It is not the best for quick iteration, exploration,
or tasks where the destination isn't clear yet.
