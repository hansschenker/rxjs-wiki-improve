// counter Erik Meijer

Great question. This is where Meijer’s position gets really sharp — and where it deserves to be pushed back.

Below is a clean contrast: Meijer’s thesis vs the strongest counter-arguments, without strawmen and without hype.

1. Translation can’t fully disappear
Meijer’s view

Developers are translators.
AI removes the need for translation.

Counter-argument

Some translation is irreducible:

Users often don’t know what they want

Requirements emerge through implementation

Constraints are discovered only when systems fail

In practice:

Intent ⇄ Feedback ⇄ Constraint


is a loop, not a one-shot description.

Developers don’t just translate — they co-discover the problem.

AI can help explore, but cannot remove:

ambiguity resolution

tradeoff negotiation

responsibility for consequences

2. Problem framing is itself a hard skill
Meijer’s view

Domain experts can express intent directly.

Counter-argument

Many domain experts:

think locally, not systemically

optimize for their slice, not the whole

under-specify failure modes

Developers often act as:

systems thinkers

boundary enforcers

integrators across domains

AI may generate solutions,
but someone still has to decide:

Which problem is worth solving?

3. Natural language is not a stable interface
Meijer’s view

Future programming languages will look like structured natural language.

Counter-argument

Natural language is:

context-sensitive

culturally biased

unstable over time

Code exists partly because:

it freezes meaning

it survives personnel change

it supports auditability

AI-generated intent systems risk becoming:

opaque

irreproducible

hard to version

Developers today provide institutional memory.
That role does not vanish easily.

4. Correctness and accountability don’t vanish
Meijer’s view

AI tolerates ambiguity and iterates.

Counter-argument

Many domains require:

deterministic behavior

provable guarantees

legal accountability

Examples:

finance

medical devices

infrastructure

safety-critical systems

If an AI system fails:

Who is liable?

Who signs off?

Who certifies correctness?

That accountability pressure creates human roles, even if implementation is automated.

5. Frontend disappearing first is plausible — but incomplete
Meijer’s view

Frontend work disappears early.

Counter-argument

Frontend is also where:

human values surface

accessibility is negotiated

ethics, persuasion, trust, and bias live

These are not purely technical problems.
They are socio-technical.

AI can generate UIs,
but deciding what is acceptable is not an algorithmic question.

6. Historical analogies can mislead
Meijer’s view

Like hand-milking or factories — roles disappear.

Counter-argument

Some roles don’t disappear — they reform:

Accountants didn’t vanish with spreadsheets

Architects didn’t vanish with CAD

Doctors didn’t vanish with diagnostics

Often:

Implementation shrinks, judgment expands.

Developers may shift from builders to:

reviewers

constraint designers

system curators

7. AI still needs curators and governors
Meijer’s view

Developers should automate themselves away.

Counter-argument

Even fully automated systems need:

oversight

monitoring

governance

intervention paths

Someone must:

set boundaries

enforce invariants

stop runaway behavior

These are not “coding” tasks,
but they are engineering tasks.

8. Tool builders don’t eliminate tool builders
Meijer’s view

Developers build systems that eliminate developers.

Counter-argument

Who builds:

the AI models?

the runtime infrastructure?

the guardrails?

the evaluation frameworks?

You don’t eliminate builders — you move the layer.

Fewer developers, higher leverage — yes.
Zero developers — unlikely.

9. Power concentration changes the equation
Meijer’s view

Users talk directly to AI.

Counter-argument

AI systems are:

expensive

centralized

controlled by few actors

This creates:

platform dependency

vendor lock-in

asymmetric power

Developers often act as counterweights:

open-source alternatives

portability layers

escape hatches

Removing them has political consequences.

10. The strongest synthesis (where both sides meet)

Meijer is right that:

mechanical coding shrinks

translation layers collapse

syntax loses value

Counter-arguments are right that:

responsibility remains

system thinking remains

governance remains

The likely outcome:

Fewer developers, but more responsibility per developer.
Less coding, more judgment.

Final synthesis in one paragraph

Erik Meijer correctly identifies that AI collapses the translation gap between human intent and computation, making large parts of traditional development obsolete. The strongest counter-argument is not that AI can’t do this, but that problem discovery, accountability, system boundaries, and social responsibility cannot be automated away. Developers may disappear as typists of logic, but persist as designers of constraints, curators of intent, and stewards of systems whose failure still matters.

Alright — let’s map Meijer vs counter-arguments onto an RxJS worldview where:

values flow over time

operators rewire streams

sharing + cancellation are explicit

business meaning lives in user functions, not in the transport

No philosophy needed — just stream topology.

The cast in RxJS terms
End user intent

intent$ : Observable<Intent>

messy, partial, changing

arrives over time (questions, corrections, constraints)

Computer execution

effects$ : Observable<Effect>

concrete actions: deploy, bill, render, mutate DB, send email

must be auditable and replayable

Developer (classic role)

A manual operator sitting between:
intent$ -> (human translate) -> spec$ -> code$ -> effects$

Meijer’s claim is basically:

AI deletes that manual operator.

Meijer’s thesis as stream rewiring
Old topology (developer as translator)
intent$  --(human translate)-->  spec$  --(engineer)-->  code$  -->  effects$

New topology (AI collapses translation)
intent$  --(AI compile)-->  effects$


In RxJS language:

Developers used to be a custom operator (translation + compilation)

AI becomes a built-in operator that maps intent to execution

So Meijer is saying:

“We remove a whole pipe segment. Fewer hops, less distortion.”

That matches your earlier “lost in translation” point: each hop is a lossy transform.

The counter-arguments as RxJS invariants

The pushback is basically:

“Even if you remove the translator, the pipeline still needs guards, contracts, replay, and cancellation.”

That’s very RxJS.

1) Translation doesn’t vanish because the source is unstable

intent$ is not a single value; it’s a stream with revisions.

So the real pipeline is:

const intent$ = merge(
  initialRequest$,
  clarifications$,
  constraintChanges$,
  "no, not that" corrections$
);


That means execution must be governed by a flattening policy:

switchMap: only latest intent matters (interactive iteration)

concatMap: queue changes (regulated workflows)

exhaustMap: ignore while busy (safety / lockouts)

mergeMap: allow overlap (batch exploration)

Counter-argument: you can’t “just compile once.” You must pick a policy.

2) Accountability = materialize + log + replay

In high-stakes systems you need:

what happened

why it happened

what intent produced it

what model/version produced it

That’s literally:

effects$.pipe(
  materialize(),       // turn next/error/complete into values
  timestamp(),         // make time explicit
  tap(writeAuditLog),  // persist
  shareReplay(1)       // current truth for inspectors
);


Counter-argument: AI can generate effects, but someone must build/own the audit stream.

3) Natural language is a noisy source

Free-form prompts are like event streams with:

duplicates

out-of-order corrections

ambiguity spikes

partial data

So you need classic stream hygiene:

intent$.pipe(
  debounceTime(300),            // stop thrashing
  distinctUntilChanged(eqIntent),
  filter(isSufficientlySpecified),
  timeout({ first: 10_000 }),   // don’t hang forever
);


Counter-argument: the “language interface” is not stable; the pipeline must stabilize it.

4) Correctness boundaries are gates

For compilers/OS/finance/medical: you need hard gates.

In RxJS, that’s a validation stream that must be true before effects flow:

const okToExecute$ = combineLatest([intent$, policy$, constraints$]).pipe(
  map(validateAll),
  shareReplay(1)
);

const guardedEffects$ = okToExecute$.pipe(
  switchMap(ok => ok ? effects$ : EMPTY)
);


Counter-argument: someone owns validateAll and the constraint model. That role survives.

introduce defaults

intent$ -> map(humanInterpretation) -> map(devInterpretation) -> effects$

Still a map, but fewer manual transforms.

That’s the RxJS split you like:

Dataflow layer: what it means

Orchestration layer: how/when it runs

AI helps mostly with dataflow generation, but orchestration still must be explicit.

If classic devs were writing the business transforms, then AI eats that.

What remains is building the machine:
