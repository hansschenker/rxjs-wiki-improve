Here are the main takeaways from this chat (and from John Hughes’s segment) that you can reuse as “rules of thumb”:

The real world can’t be forked or rolled back. Any effect model that lets you branch reality, try both branches, then keep the nicer outcome is fundamentally incompatible with actual IO.

IO works by enforcing a single “world line.” 
Conceptually: one world-state in → one world-state out, step by step. The whole point is preventing duplication and discarding of “the world”.

The crucial trick is the abstraction barrier. Even if IO behaves like World -> (A, World), programmers must not be able to touch the world token. Only composition is allowed; execution is reserved for the runtime.

Stop trying to “unwrap” effects. The beginner impulse (“I have IO a but want a”) is exactly what the type system blocks. The correct move is always: keep composing inside the effect (via bind / sequencing).

Monads are two-layered: common wiring + special meaning.

return/bind = standardized connectors (plumbing)

“extra ops” (fail/tick/random/print/…) = the actual reason you chose that monad
This split is why a single interface can serve many domains.

Standard plumbing buys you reusable libraries. Helpers like “lift a pure function over computations” and “sequence a list of computations” aren’t about IO specifically—they’re reusable wiring patterns that work across many effects.

Monad laws are refactoring guarantees. Left/right identity + associativity are there so you can regroup, extract helpers, inline helpers, and rebracket long sequences without silently changing behavior.

In real systems, ‘equality’ often means ‘equivalent enough’. Bottom/partiality and randomness highlight that you sometimes need an observational notion of sameness (e.g., “same distribution” rather than “same seed → same outputs”). Being explicit about equivalence prevents fake “law-breaking” arguments.

Property-based testing is a practical safety net. QuickCheck-style tests (including generated “function-like” values and shrinking counterexamples) are great for validating your effect instances / combinators when proofs are overkill.

Design lesson you can reuse outside Haskell (including FRP):
Define (1) wiring combinators (generic, reusable) and (2) domain primitives (what actually happens), then protect invariants with an abstraction barrier and tests. That’s the pattern that scales.

If you want, I can turn these into a small checklist you can apply when you invent a new “effect wrapper” (e.g., a domain-specific computation type or a DSL of combinators): invariants, equivalence, laws, and tests.

I/O: Only composition is allowed; execution is reserved for the runtime.

Yes — that sentence is basically the contract that makes Haskell-style IO workable:

What it means

Composition is allowed: in your program you’re allowed to build bigger I/O programs out of smaller ones.

In Haskell that’s do / (>>=) / return and library combinators (liftM2, sequence, traverse, …).

You’re assembling a recipe.

Execution is reserved for the runtime: only the runtime system is allowed to actually run that recipe against the real world.

That’s why your main has type IO (): it doesn’t “do IO” in pure code; it hands an IO action to the runtime.

Why this restriction exists

If user code could execute an IO a whenever it wanted, it could also:

run the same IO action twice on the “same world” (duplicate reality),

choose between outcomes (“try both and pick the nicer universe”),

discard a world transition (“undo” effects by dropping the new world).

So Haskell blocks the one operation that would break everything:

runIO :: IO a -> a (this does not exist in safe code)

What you do instead

You keep the value inside IO and continue composing:

-- You can’t do: a = runIO (readFile "x")
-- You do:
main :: IO ()
main =
  readFile "x" >>= \contents ->
  print (length contents)

Or with do:

main :: IO ()
main = do
  contents <- readFile "x"
  print (length contents)

Notice the pattern:

the contents exists only after the runtime runs readFile,

and you only get to use it by wiring the next step (print …) in the same IO chain.

The hidden “world token” intuition (the reason it works)

Conceptually, IO a behaves like:

-- mental model only
IO a ≈ World -> (a, World)

But the World token is not accessible to you, so you can’t fork it, reuse it, or throw it away. bind is the safe connector that threads it forward exactly once.

One important footnote

There are “escape hatches” like unsafePerformIO, but the name is honest: it bypasses this contract and can break referential transparency and reasoning guarantees.

This is the same shape as:

Observable: composition builds a dataflow description

Subscription: the outside world “presses play” and actually runs it

In both cases: you don’t “extract the value”; you wire what happens when it arrives, and the execution boundary is explicit.

The core move

You go from:

Practical refactoring rules

When you see IO a and want to “use the a”:

Pure transform? Use a “map-like” step

“when I eventually get a, compute f a”
(still in IO; no new effects)

Need another effect based on a? Use bind/chain

“when I get a, choose the next IO action”

Many actions? Use “sequence-like” wiring

“run these actions in order and collect results”

Why this is the safety guarantee

If you could unwrap IO a, you could duplicate reality, branch timelines, discard effects, or replay actions. Keeping composition inside IO prevents all of that: there’s only one world-line, and you can only move forward by wiring the next step.

Layer 1: common wiring (the plumbing)

This layer answers only: how do computations connect over time?

return: inject a plain value into the computation context (no extra effect)

bind (>>=): run one computation, take its produced value, choose the next computation

Nothing here says what your effect is. It only standardizes sequencing and value handoff.

So return/bind give you a universal “pipeline connector” for any effectful world.

Layer 2: special meaning (the reason you chose it)

This layer answers: what can happen in this domain that pure code can’t express?

Examples:

Maybe: nothing = failure / early exit

State: get/put/tick = read/update hidden state

Random: generate = sample randomness

IO: print/readFile/http = interact with the real world

These are the domain primitives.

Why this split is powerful

Because once the wiring is uniform, you get all these benefits for free:

Generic libraries: sequence, traverse, liftM2, foldM, replicateM work for every monad

Generic syntax: do notation targets the same plumbing everywhere

Uniform refactoring laws: identity + associativity explain why regrouping “steps” is safe

Domain clarity: your effect is visible exactly where you call the special ops; the rest reads like normal composition

Operators like map, mergeMap, switchMap, concatMap, exhaustMap are the plumbing: they rewire streams.

Your sources/effects (fromFetch, DOM events, timers, WebSocket, filesystem bridge, etc.) are the special meaning: why this stream exists at all.

So: one wiring API, many domains — because the wiring stays stable while the domain primitives change.

Thought for 14s

Exactly. Those two phrases are wiring shapes, not “IO tricks”.

1) “Lift a pure function over computations” = combine results without leaving the effect

You have a pure function:

f :: a -> b -> c

…but you don’t have a and b directly — you have computations that will produce them:

ma :: m a

mb :: m b

Lifting means: run the computations using the monad’s rules, then apply f to the produced values, staying inside m.

What changes per effect is what “run” means:

Maybe: “run” may fail → if either is Nothing, the whole thing is Nothing

State: “run” threads state → mb sees the state left by ma

IO: “run” performs effects → ma happens, then mb happens, then you combine the values

So the shape is reusable; only the effect’s meaning changes.

2) “Sequence a list of computations” = turn many steps into one step

You start with:

[m a] (a list of computations)

and you want:

m [a] (one computation that produces a list)

Sequencing means: “do them in order (per monad rules), collect the results.”

Again, different meaning per effect:

Maybe: stop at first failure

State: thread state through the whole list

IO: perform effects in order, gather results

This is why it’s so powerful: you can write “run these steps and collect results” once, and it works everywhere.

The same idea in RxJS terms

“lift a pure function over computations” ⇢ combine streams, then map with a pure function

wiring: combineLatestWith / withLatestFrom / zipWith

meaning lives in the user function (f)

“sequence a list of computations” ⇢ choose a flattening policy for a list of streams

sequential: concat(...sources) (queue)

parallel: forkJoin(sources) (wait for all)

overlap: merge(...sources) (allow overlap)

Same pattern: generic wiring + domain-specific effect semantics.

Code last

Haskell – lift & sequence

-- liftM2: combine two computations using a pure function
liftM2 :: Monad m => (a -> b -> c) -> m a -> m b -> m c
liftM2 f ma mb =
  ma >>= \a ->
  mb >>= \b ->
  return (f a b)

-- sequence: run a list of computations and collect results
sequence :: Monad m => [m a] -> m [a]
sequence []     = return []
sequence (m:ms) = liftM2 (:) m (sequence ms)

RxJS – analogous wiring

import { combineLatestWith, map, concat, forkJoin, merge } from "rxjs";

// lift a pure function over two computations (streams)
const c$ = a$.pipe(
  combineLatestWith(b$),
  map(([a, b]) => f(a, b))  // f = business logic; operators = wiring
);

// sequence a list of computations (streams): pick a policy
const sequential$ = concat(...sources);     // queue
const parallel$   = forkJoin(sources);      // all done -> array
const overlap$    = merge(...sources);      // allow overlap

Thought for 21s

Yes. That’s the whole superpower of FRP/Rx-style thinking:

Operators are the wiring grammar (how things flow, combine, cancel, share, and sequence over time).

Domains are just payload + naming (what the values mean and which side effects you trigger when they arrive).

So the domain can change completely, while the operator moves stay identical.

What stays the same

These are the “forever moves”:

Shape changes: map, scan

Gating: filter, takeWhile, takeUntil

Joining time: withLatestFrom, combineLatest, zip

Flattening policy:
mergeMap (overlap), switchMap (latest), concatMap (queue), exhaustMap (ignore while busy)

Error channel choices: catchError, retry, materialize/dematerialize

Make lifecycle explicit: teardown with takeUntil(...)

Make sharing explicit: share, shareReplay(1) (only when you truly want hot)

What changes per domain

Only these things:

Sources (where time/value enter): UI events, HTTP, timers, websockets, DB, animations…

Your functions (business meaning): reducers, validators, mappers, renderers

Your vocabulary (custom operators): whenSaveRequested, startCheckout, animateTo, canStartSave, etc.

The clean pattern

Keep operators as plumbing, push domain meaning into user functions and domain-named wrappers:

operators rewire streams

user functions decide meaning

domain operators are just named wiring bundles

import { merge, of, Subject } from "rxjs";
import { map, scan, shareReplay, switchMap, catchError, takeUntil } from "rxjs/operators";

// domain-specific: messages
type Msg =
  | { type: "Requested" }
  | { type: "Succeeded"; data: unknown }
  | { type: "Failed"; error: unknown };

// domain-specific: state + reducer
type Model = { status: "idle" | "busy" | "ok" | "error"; data?: unknown; error?: unknown };
const init: Model = { status: "idle" };

const update = (m: Model, msg: Msg): Model => {
  switch (msg.type) {
    case "Requested": return { status: "busy" };
    case "Succeeded": return { status: "ok", data: msg.data };
    case "Failed": return { status: "error", error: msg.error };
  }
};

const destroy$ = new Subject<void>();

// domain-specific sources
const intent$ = requestClick$.pipe(map(() => ({ type: "Requested" } as const)));

const effects$ = intent$.pipe(
  // plumbing policy: only latest request wins
  switchMap(() =>
    apiCall$().pipe(
      map((data) => ({ type: "Succeeded", data } as const)),
      catchError((error) => of({ type: "Failed", error } as const))
    )
  )
);

// plumbing: unify all messages, reduce to state, share explicitly
const messages$ = merge(intent$, effects$);

const model$ = messages$.pipe(
  scan(update, init),
  shareReplay({ bufferSize: 1, refCount: true }),
  takeUntil(destroy$)
);
