The clean way to understand all of these is this:

a cold source describes work, and every subscription would normally start that work again. Multicasting inserts a Subject-like connector in the middle so multiple downstream subscribers can watch the same upstream execution instead of each creating their own one. In current RxJS, the modern center of gravity is share and shareReplay; the older multicast/publish* APIs are deprecated and slated for removal in v8.

So the real questions are not “which operator is magic?” but:

What connector sits in the middle? Subject, BehaviorSubject, ReplaySubject, or AsyncSubject?
Who starts the shared execution? An explicit connect(), or the first subscriber automatically?
When does the shared execution reset? On error, on complete, or when subscriber count drops to zero?
1) multicast: the old general-purpose multicasting operator

multicast is the old low-level “bring your own Subject” operator. It creates a ConnectableObservable that uses the provided subject to fan out source values to multiple consumers. It is deprecated, and the docs say to use connectable, connect, or share instead. When multicast is used with a selector, the migration guide says to replace that usage with connect.

What flows over time with multicast

Step by step:

You start with a cold source. Nothing runs yet.
multicast places a Subject-like connector between source and consumers.
Downstream subscribers attach to the connector, not directly to the source.
Still nothing runs until the connectable stream is connected.
When connect() happens, the source subscribes once into that connector.
Each new source value is pushed through the connector to all current subscribers.

That means multicast separates two concerns very explicitly:

wiring the shared graph
starting the shared graph

This is why it used to matter so much historically: it gave you precise control over when the shared execution begins.

Why it was useful

It let you pick the connector yourself. If you passed a Subject, late subscribers saw only live values. If you passed a ReplaySubject, late subscribers could receive buffered history. If you passed a BehaviorSubject, they got current state immediately. If you passed an AsyncSubject, they got only the last value on completion. That connector choice is the whole behavioral core of multicasting.

Why it is deprecated

Because RxJS now splits the old use cases more cleanly:

use connectable when you want explicit connect()
use connect when you want the old selector-style multicasting
use share when you want automatic connect/disconnect with ref counting

So multicast is best understood today as the ancestor API that exposed the core mechanism directly.

2) The publish* family: named presets of multicast

The publish* operators are basically old convenience presets around multicasting. They are also deprecated. Each one chooses a different connector and therefore a different “late subscriber” policy.

2.1 publish()

publish() gives you a connectable stream that multicasts through a single underlying Subject instance, and it does not start until you call connect(). The docs describe share as an alias for multicast(() => new Subject()), refCount(), which is the automatic-start cousin of this old pattern.

Practical meaning:

current subscribers share one live execution
late subscribers do not get history
start is explicit unless you add refCount() afterward

So publish() is “share live values, no replay, explicit start”.

2.2 publishBehavior(initialValue)

This uses a BehaviorSubject under the hood. The docs say source.pipe(publishBehavior(initValue)) is equivalent to connectable(source, { connector: () => new BehaviorSubject(initValue), resetOnDisconnect: false }). A BehaviorSubject requires an initial value and emits its current value whenever it is subscribed to.

Practical meaning:

the shared stream always has a current value
a late subscriber immediately gets that current value
then it continues receiving new live values

This is why publishBehavior felt like “multicast state” rather than just “multicast events”.

2.3 publishReplay(bufferSize, windowTime?)

This is the history-preserving variant. The migration docs describe replacing publishReplay with connect or share using a ReplaySubject connector. A ReplaySubject replays old values to new subscribers.

Practical meaning:

current subscribers share one execution
late subscribers can receive buffered past values
how much history they get depends on bufferSize and optionally windowTime

This is the old “cache the latest N values” tool.

There is one subtle but important migration wrinkle: RxJS maintainers discussed that some recommended share replacements for publishReplay(...).refCount() were not behaviorally identical in every case. In particular, the exact reset semantics and connector reuse matter. So when migrating old publishReplay + refCount code, you need to be very deliberate about reset behavior.

2.4 publishLast()

This is the completion-gated variant. The docs say publishLast() is equivalent to connectable(source, { connector: () => new AsyncSubject(), resetOnDisconnect: false }). An AsyncSubject emits only its latest value, and only when it completes.

Practical meaning:

subscribers share one execution
nobody gets intermediate values
everyone gets the final value only when the source completes

So publishLast() is “multicast the result”, not “multicast the live stream”.

What the whole publish* family is really teaching

The family is valuable because it makes the connector policy visible:

publish → Subject → live only
publishBehavior → BehaviorSubject → current value
publishReplay → ReplaySubject → past buffered values
publishLast → AsyncSubject → final value on completion

That is the deepest lesson: multicasting is mostly about connector choice plus start/reset policy.

3) share: the modern operator for automatic multicasting

share is the current main operator for “make this cold source shared”. It returns a new Observable that multicasts the original source, and while there is at least one subscriber it stays subscribed upstream. In older docs language, it corresponds to the classic multicast(() => new Subject()), refCount() idea, but modern share is more configurable than that.

What flows over time with share

Step by step:

First subscriber arrives.
share creates its connector. By default that connector is a plain Subject.
It subscribes that connector to the source.
Source values flow into the connector and fan out to all current subscribers.
More subscribers join the same connector and therefore the same source execution.
When the last subscriber leaves, share can tear down and reset, depending on config.

So share is “automatic connect + automatic disconnect + configurable reset”.

The default share config

The current source shows these defaults:

connector = () => new Subject()
resetOnError = true
resetOnComplete = true
resetOnRefCountZero = true

That default means:

use a plain live-only subject
if the source errors, forget the old shared state
if the source completes, forget the old shared state
if everyone unsubscribes before completion/error, tear down and go back to a cold-like state

In practical terms, default share() is excellent for “share a live execution while someone is watching, then clean up”.

share config, one option at a time
connector

connector is the factory that creates the subject used to multicast to consumers. This is the most important option because it decides what late subscribers see.

Use it like a policy switch:

Subject → live-only sharing
BehaviorSubject(seed) → current-state sharing
ReplaySubject(n) → replay/cache sharing
AsyncSubject() → final-result sharing

So share can emulate most of the old publish* family just by changing connector.

resetOnError

The docs expose resetOnError as either a boolean or a function returning an ObservableInput. In the implementation, true resets immediately, false does not reset, and a function delays reset until the notifier emits.

Practical meaning:

true: after an error, the shared machine is discarded; next subscriber gets a fresh machine
false: keep the errored shared state
function: postpone the reset, which gives you cooldown/retry-window behavior
resetOnComplete

Same shape: boolean or notifier factory. The docs say if true, completion resets internal state back to a cold-like state; the source implementation again shows immediate reset for true, no reset for false, delayed reset for a notifier.

Practical meaning:

true: after completion, later subscribers start a fresh execution
false: keep the completed connector state around
function: keep the completed state for a while, then reset later
resetOnRefCountZero

This is the “nobody is listening now” policy. The docs expose it as boolean or notifier factory; the implementation shows it triggers when subscriber count reaches zero due to unsubscription, and again supports immediate, never, or delayed reset.

Practical meaning:

true: when the last subscriber leaves, disconnect and reset
false: keep the shared connection/connector state alive even with zero subscribers
function: wait for some external timer/signal before resetting, so a quick resubscribe can reuse the same execution

This option is often the most operationally important one in UI apps.

Why share is the modern default

Because it makes time, lifetime, and sharing explicit without forcing manual connect(). You say:

what connector is in the middle
what should happen when the source errors/completes
what should happen when listener count falls to zero

That is more expressive than the old publish* + refCount combinations.

4) shareReplay: replay-oriented sharing with specific defaults

shareReplay is not a separate multicasting universe. The RxJS docs call it a specialization around replay, and the current implementation literally builds it on top of share with a ReplaySubject connector.

The current implementation is effectively:

connector: () => new ReplaySubject(bufferSize, windowTime, scheduler)
resetOnError: true
resetOnComplete: false
resetOnRefCountZero: refCount

That means shareReplay is really “share, but with replay semantics and a different reset policy”.

What shareReplay does operationally

Step by step:

First subscriber arrives.
A ReplaySubject is created.
The source is subscribed once, and its values are pushed into that replay buffer.
Current subscribers get live values.
Late subscribers immediately receive buffered history, then continue with live values.

That is why shareReplay feels like “shared cache over time”.

The most important shareReplay choice: refCount

The docs are very explicit here.

By default, shareReplay uses refCount: false. That means when subscriber count drops to zero, it does not unsubscribe the source; the inner ReplaySubject stays connected, and the source can keep running forever. This default exists because shareReplay is often used for expensive setup that you do not want to restart.

If refCount: true, then when subscriber count reaches zero, the source is unsubscribed, the replay subject is unsubscribed, and a later subscriber will get a new ReplaySubject and therefore a new source subscription.

So:

shareReplay({ bufferSize: 1, refCount: false }) = cache and keep alive
shareReplay({ bufferSize: 1, refCount: true }) = cache while shared, but tear down when nobody is listening
Error and completion behavior of shareReplay

The current docs/source say:

a successfully completed source stays cached forever
an errored source can be retried
implementation uses resetOnError: true and resetOnComplete: false

So after completion, the replayed final buffer remains available to late subscribers. After error, the shared replay state resets so a future subscription can create a fresh run.

That makes shareReplay excellent for “remember successful result”, but it also means you must choose refCount carefully for long-lived or infinite sources.

5) The big mental model difference between old and new APIs

The old APIs made you think mostly in terms of connectable observables:

create connectable thing
subscribe consumers
call connect() when ready
maybe add refCount() to automate connection management

The modern APIs make you think mostly in terms of sharing policy:

what subject shape is in the middle?
when should shared state reset?
should zero subscribers stop the source or keep it alive?

That is a better fit for most app code.

6) When to use what

Use connectable / connect when you truly need an explicit connection boundary, which is the real modern replacement for the spirit of multicast and publish*.

Use share() when you want plain shared execution with automatic lifecycle management and little or no replay. This is the normal default for shared live streams.

Use share({ connector: ... }) when you want custom sharing semantics that match old publishBehavior, publishReplay, or publishLast, but with explicit reset policy.

Use shareReplay() when replay is the point: you want late subscribers to receive previously emitted values. Then make refCount an explicit choice, because that single flag decides whether the source stays alive with zero subscribers.

8) The shortest correct summary

multicast is the old raw mechanism.
publish* is the old family of preset connectors.
share is the modern automatic multicasting operator with explicit reset controls.
shareReplay is share with a ReplaySubject and replay-oriented defaults.

The real machine is always the same:

source execution → connector subject → multiple downstream subscribers

What changes is:

whether the connector is live/current/replay/final
whether connection is manual or automatic
whether the machine resets on error, completion, or zero listeners

The single most common production mistake is this:

using shareReplay(1) without thinking about refCount, and accidentally keeping an infinite source alive after all subscribers are gone. The docs explicitly warn that the default refCount: false means the source will not be unsubscribed when ref count drops to zero.

The second most common mistake is assuming an old publishReplay(...).refCount() migration to share(...) is automatically identical. RxJS maintainers discussed that some recommended replacements were not exact behavioral matches, especially around replay connector reuse and reset semantics.
