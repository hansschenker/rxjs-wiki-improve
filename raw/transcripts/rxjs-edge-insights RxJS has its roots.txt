RxJS has its roots in Reactive Extensions for .NET (Rx.NET), created at Microsoft in 2009 by Erik Meijer and colleagues. It began with one key insight: IObservable is the dual of IEnumerable, meaning asynchronous push‑based sequences can be treated like synchronous pull‑based collections. 

Below is a clear, structured explanation of the origins of RxJS, grounded in the authoritative sources we found.

🌿 The Deep Roots of RxJS
1. Reactive Extensions (Rx.NET), 2009 — The Birthplace
RxJS originates directly from Reactive Extensions for .NET, created at Microsoft Research in 2009.
The foundational idea was:

IEnumerable → pull values (you ask for the next item)

IObservable → push values (the system sends you items)

This duality allowed asynchronous events to be treated like collections, enabling LINQ‑style operators over time‑based data.

This is the conceptual seed from which RxJS grew.

2. The Observer + Iterator + Functional Fusion
Reactive Extensions unified three ideas:

Observer pattern (push notifications)

Iterator pattern (sequences of values)

Functional operators (map, filter, reduce)

This combination became the blueprint for all later ReactiveX implementations, including RxJS.

3. RxJS 1.x (2012) — The First JavaScript Port
In 2012, Matthew Podwysocki and Bart de Smet ported Rx.NET to JavaScript.
Key characteristics:

Mirrored the Rx.NET API closely

Used a global Rx.Observable namespace

Introduced Observables, Observers, Schedulers, and Subjects to JS

This was the moment RxJS entered the JavaScript ecosystem.

4. ReactiveX Becomes a Cross‑Language Standard
ReactiveX expanded into many languages (Java, Python, C++, JS, etc.), all sharing the same conceptual model:

Observables emit next, error, and complete events

Observers subscribe to these emissions

Operators transform sequences

RxJS is the JavaScript implementation of this broader ReactiveX family.

5. Angular Adoption (2014–2016) — The Explosion
Angular 2 adopted RxJS as its core async primitive for:

HTTP

routing

event streams

change handling

This drove massive adoption across the JS ecosystem.

6. RxJS 5 Rewrite (2016) — Modern Architecture
Ben Lesh led a full rewrite of RxJS:

major performance improvements

introduction of Schedulers

preparation for pipeable operators

This version made RxJS fast and production‑ready.

7. RxJS 6 (2018) — Pipeable Operators
The biggest shift:

operators moved out of the prototype chain

pipe() became the standard composition model

tree‑shaking became possible

This is the RxJS we know today.
