Stream-native framework thesis
A stream-native framework treats time-varying values, events, and async resources as the same thing:

State: a stream of values (often with a “current value” notion)

Events: a stream of discrete emissions

Async: a stream representing lifecycle, cancellation, retries, concurrency

UI: a stream-driven projection from app streams to DOM updates

The promise is a single mental model: compose behavior, don’t orchestrate callbacks.

What it would look like per web feature
Routing as streams
Routing becomes a set of streams:

Route state: route$ : Observable<Route> (path params, query, fragment)

Navigation intents: navigate$ : Observable<NavCommand>

Guards/resolvers: higher-order streams with explicit cancellation semantics

guards: Observable<boolean> (or richer Decision)

resolvers: Observable<Data>

Transitions: defined by a flattening policy:

switchMap for “latest navigation wins”

concatMap for “queue navigations”

exhaustMap for “ignore while navigating”

Routing stops being “a service that mutates state” and becomes pure stream calculus + policy choice.

Web requests as streams
HTTP becomes a first-class stream resource:

Request spec stream: req$ : Observable<RequestSpec>

Response stream: res$ : req$.pipe(switchMap(http))

Caching & dedupe: shareReplay, keyed multicasting, invalidation streams

Cancellation: unsubscribe = abort (wired to AbortController)

Retries/backoff: expressed as stream operators, not ad-hoc middleware

This makes concurrency semantics explicit (again: switchMap vs mergeMap is architecture).

Templates and databinding as stream projection
A template is a function from streams to a renderable structure:

View model streams: vm$ : Observable<VM>

Binding: derived streams vm$.pipe(map(vm => vm.x))

Events: DOM events as streams feeding into reducers/effects

Rendering: a reconciler that consumes:

“current value” for initial paint

incremental updates thereafter

The key decision: push rendering (re-render on emissions) vs pull rendering (signals-like). A stream-native framework can still be efficient if it defines a “current value” protocol.

Component architecture as stream transducers
A component becomes a transducer:

Inputs: props$

Outputs: events$

State: derived, not stored (or stored as a Behavior-like stream)

Lifecycle: mount/unmount are streams too (resource scoping)

Composition becomes algebraic: components are composable functions between stream interfaces.

Fine-grained reactivity based on streams

Synchronous read of current value (for DOM patching without waiting)

Dependency tracking (so only affected bindings update)

dependency graph at binding granularity

stream operators remain the authoring surface; the runtime ensures minimal DOM work

Teaching the algebra gradually
1. Start with a single, stable mental model
The first thing a developer should learn is one sentence that stays true forever:

Label: Core model
“UI is a projection of time‑varying values; interactions create events; both are streams.”

Everything else is refinement of that sentence, not a replacement.

What you avoid early
hot vs cold discourse

schedulers

higher‑order flattening

completion semantics

exotic operators

Pick a minimal basis that covers 80% of UI needs and has clean laws.

Label: Value projection
map

Label: Filtering
filter

Label: Accumulation
scan

Label: Combining state
combineLatest (or a safer combine)

Label: Lifetime
takeUntil (scoped teardown)

That’s it.

Why a basis matters
A basis gives you a closed world:

few concepts

strong equivalences

easy reasoning

consistent code style

Everything outside the basis is framed as a derived operator or an “advanced module.”

3. Separate two stream kinds from day one: Value vs Event
A major learning cliff in RxJS is treating “current state” and “events” as the same thing.

So the framework should teach two roles first:

Label: ValueStream
time‑varying value with a current snapshot (UI bindings)

Label: EventStream
discrete occurrences (clicks, requests, messages)

This one distinction prevents a lot of confusion and immediately clarifies why some operators belong where.

Pedagogical win
Users can write good code without understanding:

cold observables

subjects

replay semantics

Because ValueStream provides “current value” by construction, and EventStream is never read synchronously.

If the framework requires people to manually manage subscribe() + teardown early, they’ll learn the wrong thing: “RxJS is fragile.”

Label: Rule
“You never subscribe() in app code.”

Label: Mechanism
Effects and bindings are scoped; teardown is automatic.

So learners start with algebraic composition, not resource management. Later you can teach the underlying mechanics as “how the runtime works.”

5. Introduce laws before introducing power
Teach equivalences early because they create confidence.

Example: map fusion

Label: Law
map(f) ∘ map(g) ≡ map(f ∘ g)

This is not academic; it explains refactoring, readability, and performance intuition (“pipelines compose”).

Do the same with 2–3 more “everyday laws”:

filter(p) ∘ filter(q) ≡ filter(x => p(x) && q(x))

takeUntil(kill$) as a lifetime boundary

“pure projection” vs “effects”

Only after people internalize a few laws do you expand the operator surface.

Higher‑order streams (Observable<Observable<T>>) are the moment many learners fall off.

So the framework should not start by teaching switchMap as “map + flatten.” It should start with named policies:

Label: Latest wins
latest(request$, http) (internally switchMap)

Label: Queue
queue(request$, http) (internally concatMap)

Label: Parallel
parallel(request$, http) (internally mergeMap)

Label: First wins
first(request$, http) (internally exhaustMap)

Then, later, you reveal:

these are all instances of “map to inner stream + flatten”

and the differences are precisely the concurrency laws you already taught

This turns “operator zoo” into “same shape, different policy.”

Cycle.js  was right about “effects as values,” but the entry cost was high.

Gradual teaching means:

keep effects explicit

keep them close to the event that causes them

keep them observable and inspectable

A practical pattern:

Label: Pure core
compute intent$, state$, vm$

Label: Effect shell
derive http$, router$, storage$ from intents

Users first learn a simple rule:

“effects come from intents”
not:

“effects are sinks in a driver architecture”

Your docs and API surface should have tiers:

Tier 1 — “Get productive”
ValueStream + EventStream

5–6 operators

Tier 3 — “Algebra”
higher‑order streams

The important part: Tier 1 code should not be “toy code.” It should scale surprisingly far.

Signals

BehaviorSubject misuse

2. EventStream<T>
Discrete occurrences over time

Purpose
Separate events from state.

Definition
An EventStream<T>:

has no current value

represents user actions, network responses, timers

is replayable for tooling

Why it’s essential
This distinction:

removes hot/cold confusion

clarifies operator intent

simplifies teaching

enables AI reasoning

Rule
ValueStreams are read.
EventStreams are reacted to.

3. Scope (Lifetime)
Algebraic time boundary

Purpose
Make component lifetimes provable, not conventional.

Definition
A Scope is:

No leaks

onDestroy

Scoped side‑effect stream

Purpose
Make effects explicit, local, and cancelable.

Definition

is always scoped

never returns values

effects are visible

tooling can inspect them

AI can reason about causality

Effects never escape their scope.

Stream‑to‑DOM binding engine

Purpose

respect Scope automatically

Why it’s minimal
no diffing

no reconciliation

no component re‑execution

just subscriptions
