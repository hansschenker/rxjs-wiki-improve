Yes—RxJS + explicit Effects can be understood as a Signal Graph Architecture, with two important nuances:

it is usually a discrete event/signal graph (streams over time), not a continuous-time “signal” system, and

the graph is often dynamic (operators like switchMap, mergeMap rewire the graph over time).

If you adopt “effects as data” (your playbook), the fit becomes especially clean: the domain graph stays pure, and effects sit at the boundary as explicit nodes.

What “Signal Graph Architecture” means in this context

A signal graph is a directed graph where:

Nodes are computations (ideally pure).

Edges are dependencies (values flow along them).

Updates propagate automatically when inputs change.

That is precisely how an RxJS pipeline behaves: sources emit, operators transform, sinks consume.

How RxJS maps to a signal graph

For a given running app:

Signals / Edges → Observable<T> instances (or “streams as wires”)

Graph nodes → operators (map, scan, combineLatest, withLatestFrom, etc.)

Inputs → event sources (DOM events, timers, sockets, storage reads)

Outputs → subscribers (render, logging, persistence writes)

Stateful signal nodes → scan, BehaviorSubject, shareReplay(1) (depending on style)

Effect nodes → effect interpreters (impure), ideally fed by a typed effect$

MODEL --> RENDER[render subscribe]


This is a signal graph where:

scan appSteps is the main state propagation node

effect$ is a typed edge that crosses into the impure world

runEffect is an impure node that returns messages back into the graph

Why “Effects-as-data” makes it more like a signal graph

If you encode effects as values (EffectRequest) and interpret them at the boundary:

The graph becomes explicitly partitioned:

Pure subgraph: deterministic transformations

Impure subgraph: I/O execution and callbacks

That mirrors classic signal-graph thinking (pure transforms inside; side-effecting sinks at edges).

Two nuances you should keep in mind
1) RxJS is often an event graph, not a “current value signal” graph

Many signal systems assume each wire has a “current value” (like spreadsheet cells or UI signals).

RxJS streams may have no current value unless you make them behave like “signals” via:

BehaviorSubject

shareReplay({ bufferSize: 1, refCount: true })

a scan-built state stream that is replayed

In practice:

Treat model$ as your “signal” by replaying the latest state.

2) The graph can be dynamic (time-varying topology)

Operators like switchMap create subgraphs per emission and cancel prior ones. That is still a graph architecture, but not a static DAG.

This matters for:

reasoning about lifecycle / cancellation

avoiding accidental resubscriptions

ensuring shared edges are explicitly multicasted (shareReplay, share)

“Signal graph” vs “dependency graph” vs “dataflow graph”

In software architecture terms, RxJS + Effects is most precisely:

a reactive dataflow graph (values propagate along edges)

optionally a signal graph when you enforce “latest value” semantics on key wires (model$, derived selectors)

with explicit effectful boundary nodes

Practical checklist to make your RxJS app behave like a clean signal graph

Make “state signals” replay latest: model$ = ...shareReplay(1)

Keep domain transitions pure: evolve(state, cmd) (no I/O, no RxJS)

Treat errors as data: Result/Either emissions, not error() termination

Keep effects explicit and typed: effect$: Observable<EffectRequest>

Interpret effects only at the boundary and feed results back as messages

Avoid hidden sharing: if multiple consumers need the same edge, use shareReplay deliberately
