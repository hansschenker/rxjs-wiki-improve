What is Rxjs
We are going to explain what RxJS is in a comprehensive manner.
RxJS (Reactive Extensions for JavaScript) is a library for reactive programming using Observables, to make it easier to compose asynchronous or callback-based code.

Let's break it down:

Reactive Programming: A programming paradigm oriented around data streams and the propagation of change. This means that it becomes possible to express static (e.g., arrays) or dynamic (e.g., event emitters) data streams with ease, and also to react to changes in those streams.

Observables: The core concept of RxJS. An Observable represents a stream of data that can be observed over time. You can think of it as a collection of future values or events that you can listen to.

Operators: RxJS provides a wide range of operators that allow you to transform, combine, manipulate, and work with the data streams in a declarative manner. Examples include map, filter, merge, switchMap, etc.

Observers and Subscriptions: An Observer is a consumer of values delivered by an Observable. A Subscription represents the execution of an Observable, and can be used to cancel the execution.

Subjects: A Subject is a special type of Observable that allows values to be multicasted to many Observers. Subjects are like EventEmitters.

Schedulers: Schedulers are used to control the timing of when notifications are delivered by an Observable.

Why use RxJS?

Handling Asynchronous Operations: It provides a unified way to handle asynchronous operations (like events, HTTP requests, timers, etc.) and even synchronous operations as streams.

Composition: The ability to compose multiple asynchronous events or operations together.

Error Handling: Provides powerful error handling mechanisms within streams.

Operators: The rich set of operators allows for complex transformations and combinations of data streams with minimal code.

Cancellation: Easy cancellation of asynchronous operations through subscriptions.

Example:

Let's say you have a button and you want to handle clicks, but you want to throttle the clicks so that you only take one click per second.

Without RxJS, you might use a flag and setTimeout. With RxJS, you can do:

javascript
import { fromEvent } from 'rxjs';
import { throttleTime } from 'rxjs/operators';

const button = document.querySelector('button');
fromEvent(button, 'click')
  .pipe(throttleTime(1000))
  .subscribe(event => console.log('Clicked!', event));
In this example, fromEvent creates an Observable from button click events. Then we use the throttleTime operator to only allow one event per second. Then we subscribe to the resulting Observable to log the event.

RxJS is widely used in Angular for handling HTTP requests, routing, and form events, but it can be used in any JavaScript environment.

Note: RxJS 6 and above use a pipeable operator approach (using pipe method) which is more tree-shakable and functional.

The Rxjs Observable is an Abstract Data Type of a Stream
We are given a statement: "The Rxjs Observable is an Abstract Data Type of a Stream"
We are to explain what this means and elaborate on the nature of RxJS Observables as streams and abstract data types.

Let's break it down:

Abstract Data Type (ADT):

In computer science, an ADT is a mathematical model for data types. It defines a set of operations on the data and the behavior of these operations, but not the implementation.

Examples: Stack, Queue, List.

Stream:

In reactive programming, a stream is a sequence of ongoing events ordered in time. It can be thought of as a flow of data that can be observed.

RxJS Observable:

An Observable is a representation of a stream of data that can be observed over time.

It is an ADT because it provides a set of operations (such as subscribe, pipe, and operators like map, filter) that allow us to work with the stream without knowing the underlying implementation of how the events are produced.

So, the statement is saying that an RxJS Observable is an abstract way to work with streams of data.

Let's elaborate:

In RxJS, an Observable is a lazy push-based collection of multiple values. It is a function that takes an observer (which has methods like next, error, and complete) and returns a way to cancel the execution (unsubscribe).

The Observable is an ADT because:

It has a defined interface (you can subscribe to it, and it produces values over time).

It is composable with operators (which are pure functions that take an Observable and return a new Observable).

It abstracts the source of the data (whether it's from an event, an array, a promise, etc.) and allows us to interact with it in a consistent way.

Example:

We can create an Observable from a click event, or from an array, or from a promise, and then we can use the same operators (like map, filter) on all of them.

This is the power of abstraction: we don't care about the underlying data source, we just care about the operations we can perform on the stream.

Let's write a code example to illustrate:

We'll create an Observable from an array and then use the map operator to transform the values.

Then we'll create an Observable from a click event and use the same map operator.

This shows that the Observable ADT allows us to treat different data sources in the same way.

However, note that the statement is a bit philosophical. In practice, an Observable is a specific implementation of the Observable pattern (which is a design pattern) and in RxJS it is implemented as a class with methods and static functions.

But the idea is that it abstracts the concept of a stream and provides a set of operations to work with streams.

Rxjs Obserable is basically a lazy, potentially inifinite Sequence of Pair of Time T and Value a: [{T,a}..]
We are going to explain the RxJS Observable as a lazy, potentially infinite sequence of pairs (time, value).
This is a more formal and precise way to think about Observables, especially from the perspective of functional reactive programming.

Let's break down:

Sequence: An Observable is a sequence of values. This means the values come in an order, one after the other.

Pair of Time and Value: Each element in the sequence is a pair (T, a), where T is the time at which the value a is emitted.

Lazy: The sequence is lazy, meaning that it doesn't produce values until someone subscribes to it (similar to how a function doesn't run until you call it).

Potentially Infinite: The sequence can go on forever (like intervals or events that keep happening).

However, note that in RxJS, the time in the pair is implicit. The Observable doesn't explicitly give you a time for each value, but you can think of each value as being associated with a moment in time (the time at which it is emitted).

Let's formalize:

Observable<T> ≈ Stream of (time: T, value: a)

But note: In practice, we don't get a tuple (time, value). Instead, we get the value, and the time can be captured by using the timestamp operator or by using new Date() at the moment of emission.

Example:

import { interval } from 'rxjs';
import { timestamp } from 'rxjs/operators';

const source$ = interval(1000).pipe(
timestamp()
);

source$.subscribe(value => {
console.log(value); // { value: 0, timestamp: 123456789 } (the timestamp is in milliseconds since epoch)
});

However, note that the original Observable (without timestamp) just emits numbers (0,1,2,...) every second. The time is implicit in the sense that we know the interval, but the value doesn't come with an explicit time.

So, when we say Observable is a sequence of (T, a), it's more of a conceptual model. In RxJS, the time is available if you capture it (like using timestamp), but by default, the Observable emits only the value.

Another point: The time T in the pair is not necessarily the wall-clock time. It can be logical time (like in some FRP systems) or it can be the time of emission. In RxJS, it's the time of emission (if you capture it).

Let's think about the lazy aspect:

const observable = new Observable(subscriber => {
console.log('Hello');
subscriber.next(1);
});

// Without subscribing, the function above doesn't run. So, no 'Hello' is printed.

observable.subscribe(); // Now, 'Hello' is printed and 1 is emitted.

And infinite:

const infiniteObservable = interval(1000); // Emits every second, forever.

So, yes, the model of Observable as a lazy, potentially infinite sequence of (time, value) pairs is accurate and useful.

However, note that an Observable can also be finite (like an HTTP request that returns one value and completes) and can have zero or one value.

Let's consider the types:

In RxJS, an Observable can emit multiple values over time. The time is continuous (or at least, the events are discrete in continuous time). But in the sequence, we only get the values at the times they are emitted.

This model is very powerful because it allows us to reason about asynchronous events in a declarative way and use functional operators (map, filter, etc.) to transform the sequence.

Example of transformation:

Let's say we have a stream of (time, value) pairs. We can:

text
1. Filter by value: only let through pairs with value > 10.
2. Filter by time: only let through pairs that are emitted at even seconds (if we capture the time).
Without capturing time, we can still use time-based operators like debounceTime, throttleTime, etc., which work on the implicit time of emission.

So, to summarize:

The Observable is a sequence of events (values) over time.

Each event can be thought of as a pair (time of emission, value).

The sequence is lazy and can be finite or infinite.

This model is the basis for functional reactive programming (FRP) and allows for powerful compositions of asynchronous operations.

Erik Meijer brought Haskell List Comprehensions to Microsoft .Net Languages in the form of LINQ (Language integrated Monad), later LINQ was enhanced to Rx.net which was ported to Javascript as Rxjs
We are going to correct and elaborate on the statement:

Erik Meijer brought Haskell List Comprehensions to Microsoft .Net Languages in the form of LINQ (Language integrated Monad), later LINQ was enhanced to Rx.net which was ported to Javascript as Rxjs.

This statement is mostly accurate but let's clarify the history and connections.

Clarification:
LINQ and Monads:

Erik Meijer and team designed LINQ (Language Integrated Query) for .NET, which was indeed influenced by Haskell's list comprehensions and monads.

In Haskell, list comprehensions are a syntactic sugar for working with lists (which are a monad). Similarly, LINQ provides a way to write queries in a declarative style, and the underlying structure is based on monads (specifically, the IEnumerable<T> interface and the SelectMany method, which is the monadic bind).

From LINQ to Rx.NET:

LINQ was initially designed for pull-based collections (like IEnumerable<T>).

However, the same pattern of operations (like Select, Where, Aggregate) can be applied to push-based collections (events/streams).

This led to the development of Rx.NET (Reactive Extensions for .NET), which extended the LINQ pattern to asynchronous and event-based programming using IObservable<T> (the dual of IEnumerable<T>).

Duality:

There's a fundamental duality between IEnumerable<T> (pull) and IObservable<T> (push).

In the pull model, the consumer requests the next item (by calling MoveNext on an enumerator). In the push model, the producer notifies the consumer when an item is available (by calling the observer's OnNext method).

This duality is expressed in the design of Rx, which provides a similar API for both models.

Port to JavaScript:

The success and utility of Rx.NET led to ports for other languages and platforms, including JavaScript (RxJS), Java (RxJava), and others.

RxJS brought the reactive programming model with Observables to JavaScript, allowing developers to handle asynchronous events as collections.

const processStream = {
  // Clean noisy data
  clean: <T>() => pipe(
    filter((x: T) => isValid(x)),
    distinctUntilChanged()
  ),

  )
};

Real-World Example: Reusable Pagination Logic
typescript
// Generic pagination operator (works with ANY list)
function paginate<T>(pageSize: number) {
  return (source$: Observable<T[]>) => 
    source$.pipe(
      scan((acc, items) => [...acc, ...items], [] as T[]),
      map(allItems => {
        const pageCount = Math.ceil(allItems.length / pageSize);
        const pages = Array.from({length: pageCount}, (_, i) =>
          allItems.slice(i * pageSize, (i + 1) * pageSize)
        );
        return { pages, currentPage: pages.length - 1 };
      })
    );
}

// Use for ANY domain:
products$.pipe(paginate(10));       // E-commerce
articles$.pipe(paginate(20));       // News feed
logEntries$.pipe(paginate(100));    // System logs

We are going to discuss how TypeScript's static typing, especially generics, enhances RxJS.
RxJS is written in TypeScript and provides rich type definitions that allow for type-safe stream operations.

Let me break down the key aspects:

Generic Observables: The Observable<T> type is generic, where T is the type of values emitted by the observable.
This allows the TypeScript compiler to check that the values are used correctly throughout the pipeline.

Type Inference in Operators: When you use operators like map, filter, etc., TypeScript can infer the types of the values
at each step, which helps in catching errors and provides better IDE support.

Operator Signatures: Each operator is defined with precise generic types, which ensures that the operations are type-safe.

Combining Streams: When combining multiple streams (e.g., with merge, combineLatest), TypeScript can infer the type of the
resulting stream.

Example:

Let's see an example of how TypeScript generics work with RxJS.

typescript
import { Observable } from 'rxjs';
import { map, filter } from 'rxjs/operators';

// An observable that emits numbers
const numberObservable: Observable<number> = new Observable(subscriber => {
  subscriber.next(1);
  subscriber.next(2);
  subscriber.next(3);
});

// Using map and filter with type inference
const squaredObservable = numberObservable.pipe(
  filter(x => x > 1),        // x is inferred as number, returns Observable<number>
  map(x => x * x)            // x is inferred as number, returns Observable<number>
);

// Now, let's say we want to map to a string
const stringObservable = numberObservable.pipe(
  map(x => `Number: ${x}`)   // x is number, returns Observable<string>
);

// TypeScript will complain if we try to use a string method on a number
numberObservable.pipe(
  map(x => x.toFixed(2))     // OK, because x is a number
);

// But if we try to do something that's not allowed on a number, TypeScript will catch it
numberObservable.pipe(
  map(x => x.toUpperCase())  // Error: Property 'toUpperCase' does not exist on type 'number'
);

// Combining streams with different types
const stringObservable2 = new Observable<string>(subscriber => {
  subscriber.next('hello');
  subscriber.next('world');
});

// Combining the latest values from two observables
import { combineLatest } from 'rxjs';

const combined = combineLatest([numberObservable, stringObservable2]);
// combined is of type Observable<[number, string]>

// Now, when we map over the combined stream, we get a tuple of the latest values
combined.pipe(
  map(([num, str]) => `${str}: ${num}`)  // num is number, str is string
).subscribe(console.log);

typescript

Rxjs Subject is a Proxy between Observable and Observer, enabling Rxjs Multicasting
We are going to discuss the role of the Subject as a proxy in RxJS multicasting.
A Subject in RxJS acts as both an Observable and an Observer. This dual nature allows it to be a proxy that can multicast events from a source to multiple subscribers.

Let me break it down:

As an Observable: A Subject can be subscribed to by multiple observers (subscribers).

As an Observer: A Subject can receive events from an Observable (by subscribing to it) and then forward those events to all of its subscribers.

This makes Subject a central point (or proxy) that can take a single stream and multicast it to multiple subscribers.

Example:

Consider a cold Observable (like an HTTP request). We want multiple subscribers to get the same result without making multiple HTTP requests.

We can use a Subject as a proxy:

The Subject subscribes to the cold Observable.

Then, multiple subscribers subscribe to the Subject.

When the cold Observable emits a value, the Subject receives it (as an Observer) and then forwards it to all its subscribers (as an Observable).

However, note that the Subject is a hot Observable. It will only broadcast events to subscribers that are subscribed at the time of the event.

Let's look at an example in code:

Step 1: Create a cold Observable (simulating an HTTP request that returns the same data to all)
Step 2: Create a Subject.
Step 3: Use the Subject to multicast the cold Observable.

But note: There are better ways to do this using operators like multicast, share, publish, etc., which internally use Subject.

However, the core idea is that Subject enables multicasting by being a proxy.

Let me show a basic example without operators, just using a Subject:

const cold$ = new Observable(subscriber => {
console.log('Making HTTP request');
subscriber.next(Math.random());
subscriber.complete();
});

const subject = new Subject();

// Subject subscribes to the cold Observable
cold$.subscribe(subject);

// Now, multiple subscribers to the subject
subject.subscribe(value => console.log('Subscriber 1: ', value));
subject.subscribe(value => console.log('Subscriber 2: ', value));

In the above, even though we have two subscribers to the subject, the cold Observable only makes one HTTP request (logs once).

However, note that the timing of subscription matters. If we subscribe the subject to the cold$ after the individual subscribers, then the individual subscribers won't get the value because the cold$ might have already emitted.

This is where the concept of a ConnectableObservable comes in, which allows us to control when the source Observable starts emitting.

Alternatively, we can use the multicast operator (which returns a ConnectableObservable) and then call connect when we are ready.

Example with multicast:

const cold$ = new Observable(subscriber => {
console.log('Making HTTP request');
subscriber.next(Math.random());
subscriber.complete();
});

const multicasted$ = cold$.pipe(
multicast(new Subject())
) as ConnectableObservable<number>;

// Then, we subscribe to the multicasted$ (which is a hot Observable) and then connect.
multicasted$.subscribe(value => console.log('Subscriber 1: ', value));
multicasted$.subscribe(value => console.log('Subscriber 2: ', value));

// Now, we connect the subject to the cold Observable, which starts the execution of cold$.
multicasted$.connect();

But note: The above example with multicast is a bit low-level. In practice, we often use share or shareReplay which handle the connection and reference counting automatically.

However, the underlying mechanism for multicasting in RxJS is the Subject.

Let me summarize:

A Subject is a special type of Observable that allows values to be multicasted to many observers.

A Subject is also an Observer, so it can subscribe to an Observable and then re-emit the values to all of its subscribers.

This is the basis of multicasting in RxJS.

Important: There are different types of Subjects:

Subject: No initial value, only emits values after subscription.

BehaviorSubject: Requires an initial value and emits the current value to new subscribers.

ReplaySubject: Can replay a specified number of previous values to new subscribers.

AsyncSubject: Only emits the last value (and only when the source completes).

Each of these can be used as a proxy for multicasting in different scenarios.

Example with BehaviorSubject:

const cold$ = new Observable(subscriber => {
console.log('Making HTTP request');
subscriber.next(Math.random());
subscriber.complete();
});

const subject = new BehaviorSubject<number | null>(null);

cold$.subscribe(subject);

// Now, multiple subscribers to the subject
subject.subscribe(value => console.log('Subscriber 1: ', value));
subject.subscribe(value => console.log('Subscriber 2: ', value));

In this case, because BehaviorSubject requires an initial value, the first value emitted by the cold$ will be the current value and will be emitted to new subscribers immediately.

However, note that the cold$ in this example emits and completes, so the BehaviorSubject will have that emitted value as its current value.

But note: The BehaviorSubject will emit the initial value (null) to any subscriber that subscribes before the cold$ emits. Then, when the cold$ emits, it will emit that new value.

So, in the above example, if we subscribe to the subject after the cold$ has completed, then the subject will have the value from cold$ and will emit that to new subscribers.

However, if we want to avoid emitting the initial null, we can use a ReplaySubject with buffer size 1 and no initial value.

Let me adjust:

const subject = new ReplaySubject<number>(1);

cold$.subscribe(subject);

// Now, if we subscribe after the cold$ has emitted, we get the value. If we subscribe before, we will get the value when it arrives and then because of the replay, we will get it again when we subscribe after? Actually, ReplaySubject will replay the last value to new subscribers.

// So, if the cold$ has already emitted, then new subscribers to the ReplaySubject will get that emitted value immediately.

In summary, the Subject (and its variants) act as a proxy between the source Observable and multiple observers, enabling multicasting.

This is a fundamental concept in RxJS for sharing a single source of data among multiple subscribers.

Subject's Internal Proxy Mechanism:

class Subject<T> implements Observable<T>, Observer<T> {
  private observers: Observer<T>[] = [];
  
  // Observer interface: Receive from source
  next(value: T): void {
    this.observers.forEach(observer => observer.next(value));
  }
  
  error(err: any): void {
    this.observers.forEach(observer => observer.error(err));
  }
  
  complete(): void {
    this.observers.forEach(observer => observer.complete());
  }
  
  // Observable interface: Distribute to subscribers
  subscribe(observer: Observer<T>): Subscription {
    this.observers.push(observer);
    return new Subscription(() => {
      const index = this.observers.indexOf(observer);
      if (index > -1) this.observers.splice(index, 1);
    });
  }
}

Rxjs Operators are High-Order Function taking an Observable as input and returning a new Observable as output and are typed with Typescript as OperatorFunction<T,R>
We are going to discuss the type signature of RxJS operators as higher-order functions.
In TypeScript, an operator function is typically defined as:
type OperatorFunction<T, R> = (source: Observable<T>) => Observable<R>

This is a higher-order function because it takes a function (or in this case, an Observable) and returns a new function (Observable).

However, note that in RxJS, operators are often implemented as factory functions that return an operator function (which is a higher-order function).

Let me break it down:

An operator is a function that takes an Observable and returns a new Observable.

The TypeScript type for this is OperatorFunction<T, R>, which is a function from Observable<T> to Observable<R>.

Since an operator function takes a function (the source Observable) and returns a function (the new Observable), it is a higher-order function.

Example of a simple operator (map):

function map<T, R>(project: (value: T) => R): OperatorFunction<T, R> {
return (source: Observable<T>) => new Observable<R>(subscriber => {
// For each value emitted by the source, apply the project function and pass to the subscriber
const subscription = source.subscribe({
next: value => subscriber.next(project(value)),
error: err => subscriber.error(err),
complete: () => subscriber.complete()
});

text
   return subscription;
 });
}

In the above, map is a function that returns an operator function (which is a higher-order function).

However, note that in RxJS, operators are used in the pipe method of an Observable:

source.pipe(map(x => x * 2))

The pipe method takes one or more operator functions and applies them to the source Observable.

Let's see the type definition for pipe:

pipe<A>(op1: OperatorFunction<T, A>): Observable<A>

pipe<A, B>(op1: OperatorFunction<T, A>, op2: OperatorFunction<A, B>): Observable<B>

... and so on for more operators.

So, when we do source.pipe(map(x => x * 2)), the map operator function is passed to pipe and is applied to source.

Therefore, we can say that RxJS operators are higher-order functions that transform Observables.

Another way to think about it:

An operator function is a function that takes an Observable and returns an Observable.

Since it takes a function (the source Observable) and returns a function (the new Observable), it is a higher-order function.

However, note that the source Observable is not exactly a function in the traditional sense, but it is an object that implements the Observable interface (which is a push-based collection that we can think of as a function that produces values over time).

But in functional programming, we often consider streams (or Observables) as functions from time to values. So, an operator that takes an Observable and returns an Observable is indeed a higher-order function.

Let me give a more formal type definition in TypeScript:

);

This is a higher-order function because map returns a function (the operator function) that takes an Observable and returns an Observable.

Rxjs pipeline creates a dependency graph with arrows in the direction of the dataflow and with arrows created from subscriptions in the inverted direction of the dataflow
We are going to break down the statement and then elaborate on it.

The statement: "Rxjs pipeline creates a dependency graph with arrows in the direction of the dataflow and with arrows created from subscriptions in the inverted direction of the dataflow."

Let me explain:

Dependency Graph in RxJS:

In RxJS, when we create an observable pipeline (using operators), we are setting up a chain of dependencies. Each operator returns a new observable that depends on the previous observable.

Arrows in the direction of the dataflow:

The data flows from the source observable through the operators to the subscriber. So, if we think of the observables as nodes, then the edges (arrows) go from the source to the next operator, and so on, until the subscriber.

Arrows created from subscriptions in the inverted direction:

However, when we subscribe, the subscription propagates upwards (against the dataflow) to the source. This is because each observable in the chain subscribes to the one before it, until the source is reached.

Let me illustrate with a simple example:

typescript
source$.pipe(
  map(x => x * 2),
  filter(x => x > 10)
).subscribe(value => console.log(value));
The dataflow:
source$ (emits value) → map (transforms) → filter (checks condition) → subscriber (logs)

But the subscription flow (when we call subscribe on the result of pipe) is in the opposite direction:

subscriber (listener) → filter (subscribes to map) → map (subscribes to source$) → source$ (starts emitting)

So, the dependency graph has two sets of arrows:

Dataflow arrows: from source to subscriber (downstream)

Subscription arrows: from subscriber to source (upstream)

This is important because it means that until we subscribe, the observables are just inert. When we subscribe, the subscription travels up to the source, and then the data starts flowing down.

Now, let's break it down further:

Setting up the pipeline (creating the graph)
When we call pipe, we are not executing anything, just composing functions that will be called when we subscribe. Each operator returns a new observable that, when subscribed to, will subscribe to the previous observable.

Subscription (activating the graph)
When we call subscribe, the following happens:

The subscriber (the object we pass to subscribe) is wrapped into a Subscriber object (if necessary) and becomes the observer for the last observable in the chain (the one returned by filter in our example).

Then, the observable returned by filter subscribes to the observable returned by map (which is the source for the filter observable).

The observable returned by map subscribes to the original source$.

This chain of subscriptions establishes the dataflow path.

Data emission (flowing through the graph)
When source$ emits a value, it passes it to the map operator, which transforms it and passes it to the filter operator, which then conditionally passes it to the original subscriber.

Visual representation
text
Dataflow (downstream):
source$ --(value)--> map --(transformed value)--> filter --(filtered value)--> subscriber

Subscription (upstream):
subscriber --(subscribe)--> filter --(subscribe)--> map --(subscribe)--> source$
Why is this important?
Understanding this two-way graph helps in debugging and in understanding how RxJS works. For example, if you have a cold observable, the subscription to the source might trigger a new resource creation (like an HTTP request) for each subscription. Also, it helps in understanding how unsubscription propagates: when you unsubscribe from the observable chain, the unsubscription travels up to the source, potentially cancelling the HTTP request (if the source is built to do so).

In terms of dependency graph
We can think of each observable as depending on the previous one (for data). But when we subscribe, we are creating a chain of subscriptions that go back to the source.

So, the statement is accurate: the pipeline creates a dependency graph with arrows (dependencies) in the direction of the dataflow (from source to subscriber) and with arrows (subscriptions) in the inverted direction (from subscriber to source) when we subscribe.

How This Works Internally:
The Subscription Chain:
typescript
// Simplified internal structure
class Observable<T> {
  pipe(...operators: OperatorFunction<any, any>[]): Observable<any> {
    return operators.reduce((prev, operator) => operator(prev), this);
  }
  
  subscribe(observer: Observer<T>): Subscription {
    // Creates subscription chain
    const subscriber = new Subscriber(observer);
    
    // This is where backward propagation happens
    // Each operator's _subscribe method calls parent's _subscribe
    const subscription = this._subscribe(subscriber);
    
    return subscription;
  }
}

// Each operator creates a new Observable that, when subscribed to,
// subscribes to its parent (backward chain)
