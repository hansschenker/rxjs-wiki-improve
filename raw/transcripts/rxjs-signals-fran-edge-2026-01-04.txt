Here’s a clear, computer‑science‑focused explanation that cuts through the noise and gets to the essence of signals.

📡 What Is a Signal in Computer Science?
In computer science and related fields, a signal is any function, value, or event that conveys information. More formally, a signal is:

A function of one or more variables (often time) that represents some information‑carrying phenomenon.

Signals can be continuous (analog) or discrete (digital). In computing, we mostly deal with digital signals, which represent information using discrete values (often binary).

🧰 What Are Signals Used For in Computer Science?
Signals show up in several major areas of computing. Here are the most important ones:

1. Communication & Networking
Signals are the foundation of all data transmission.

Electrical signals in cables

Radio waves in Wi‑Fi

Optical signals in fiber

They encode bits so devices can communicate.

2. Operating Systems (OS Signals)
In Unix-like systems, a signal is a software interrupt sent to a process.

Examples:

SIGINT — interrupt (Ctrl+C)

SIGKILL — force termination

SIGSEGV — segmentation fault

These signals notify processes about events and control their behavior.

(Note: This meaning is different from the mathematical/engineering definition.)

3. Digital Signal Processing (DSP)
Signals represent data that needs to be analyzed or transformed:

Audio waveforms

Images (2D signals)

Sensor readings

Video streams

DSP algorithms filter, compress, or transform these signals.

4. Embedded Systems & Robotics
Signals represent physical measurements:

Temperature

Acceleration

Voltage

Position

These signals are sampled, digitized, and processed by microcontrollers.

5. Machine Learning & Data Science
Many datasets are essentially signals:

Time series

Audio

Sensor logs

EEG/ECG data

ML models often treat them as sequences or functions over time.

6. Programming Languages (Synchronous/Dataflow Languages)
Some languages (e.g., SIGNAL, Lustre, Esterel) treat signals as streams of values synchronized over time.

These are used for:

Real‑time systems

Control systems

Safety‑critical software

Nice. Let’s build something that feels like FRAN but fits the reality of RxJS and the browser: a discrete-time animation DSL on top of RxJS.

I’ll aim for something you could actually drop into rxjs-all-in-one as an “animation module”.

Goals of the mini‑FRAN‑like DSL
Behavior-like: express “value over time”, not imperative steps.

Event-like: describe discrete triggers that start/stop/compose animations.

Compositional: reuse your RxJS operators, don’t fight them.

Render-friendly: easy to hook into DOM / canvas / WebGL.

Honest: explicitly discrete-time (animation frames), no pretending to be continuous.

Core semantic model
We’ll define a very small set of abstractions:

AnimationClock: stream of frame times.

Behavior\<A\>: a value that evolves over frame time.

Event\<A\>: discrete occurrences that can influence behaviors.

Animation\<A\>: a Behavior “focused” on a finite interval (or until some condition).

Conceptually:

ts
type Time = number;          // milliseconds since start of animation (or from performance.now())

type Behavior<A> = (t: Time) => A;

// Concrete implementation: Behavior sampled on each animation frame
type BehaviorStream<A> = Observable<{ t: Time; value: A }>;

type Event<A> = Observable<{ t: Time; value: A }>;
We won’t implement “true” continuous time; we’ll define it as if it were continuous, and implement it as BehaviorStream driven by animationFrames().

Step 1: Base clock and time Behavior
First, a shared clock for animations.

ts
import { animationFrames, Observable } from 'rxjs';
import { map, share } from 'rxjs/operators';

export type Time = number;

export interface Frame {
  t: Time;   // time in ms since animation start
  dt: Time;  // delta from previous frame
}

export const animationClock: Observable<Frame> = animationFrames().pipe(
  map(({ elapsed }) => ({
    t: elapsed,
    dt: 0, // we'll fill dt via scan if needed
  })),
  share()
);
If you want dt, you can add a scan, but for most behavior math, t is enough.

Step 2: From Behavior to BehaviorStream
A Behavior is a pure function of time. The DSL core is: “given a Behavior<A>, give me a stream of sampled values”.

ts
export type Behavior<A> = (t: Time) => A;

export function sampleBehavior<A>(
  behavior: Behavior<A>,
  clock: Observable<Frame> = animationClock
): Observable<{ t: Time; value: A }> {
  return clock.pipe(
    map(frame => ({
      t: frame.t,
      value: behavior(frame.t),
    }))
  );
}
This is your FRAN-like Behavior semantics: you write pure functions, RxJS samples them.

Step 4: Tweening / easing Behaviors
Let’s define a canonical FRAN-like tween (linear interpolation) and plug in easing.

ts
export interface TweenConfig {
  from: number;
  to: number;
  duration: Time;         // ms
  easing?: (u: number) => number; // u in [0,1]
}

/**
 * tween: Behavior<number>
 * Over [0, duration], moves from `from` to `to` with optional easing.
 * For t >= duration, stays at `to`. For t < 0, stays at `from`.
 */
export function tween(cfg: TweenConfig): Behavior<number> {
  const { from, to, duration, easing } = cfg;
  const delta = to - from;

  const ease = easing ?? (u => u);

  return t => {
    if (t <= 0) return from;
    if (t >= duration) return to;
    const u = t / duration;
    return from + delta * ease(u);
  };
}
Now you have a pure Behavior that describes a transition over time.

Step 5: Relative-time animations
Most animations are “start at the moment X happens, then run for N ms”.

So we need:

an Animation type

composition with event start times.

Animation type
ts
export interface Animation<A> {
  // Behavior defined over local-time tLocal (t - tStart)
  behavior: Behavior<A>;
  duration: Time; // or Infinity for open-ended behaviors
}
Running an Animation from a start time
ts
import { switchMap, takeWhile } from 'rxjs/operators';

export function runAnimation<A>(
  animation: Animation<A>,
  start$: Event<unknown>, // when to start
  clock: Observable<Frame> = animationClock
): Observable<{ t: Time; value: A }> {
  return start$.pipe(
    switchMap(startEvt => {
      const tStart = startEvt.t;
      return clock.pipe(
        map(frame => {
          const tLocal = frame.t - tStart;
          return {
            t: frame.t,
            value: animation.behavior(tLocal),
          };
        }),
        // finite animation: stop after duration
        takeWhile(sample => sample.t - tStart <= animation.duration, true)
      );
    })
  );
}
We’re treating the FRAN-like Behavior as relative-time, aligned at the event that starts the animation.

Step 8: Composing animations
You can add combinators like:

Example: andThen
ts
export function seq<A>(
  first: Animation<A>,
  second: Animation<A>
): Animation<A> {
  return {
    duration: first.duration + second.duration,
    behavior: t => {
      if (t <= first.duration) {
        return first.behavior(t);
      } else {
        return second.behavior(t - first.duration);
      }
    },
  };
}
Now you can write:

const moveRightAndBack: Animation<number> =

  );

1. Core types (strict, explicit, minimal)
ts
// animation-core.ts
import { Observable } from 'rxjs';

export type Time = number; // milliseconds

export interface Frame {
  readonly t: Time;   // elapsed since some origin
  readonly dt: Time;  // time since previous frame
}

// Pure, FRAN-style behavior: value as a function of (local) time
export type Behavior<A> = (t: Time) => A;

// Timestamped event in animation time
export interface Timed<A> {
  readonly t: Time;
  readonly value: A;
}

// FRAN-style discrete event stream
export type Event<A> = Observable<Timed<A>>;

// Sampled behavior over an animation clock
export type BehaviorStream<A> = Observable<Timed<A>>;

// Finite (or infinite) behavior tied to an interval
export interface Animation<A> {
  readonly duration: Time;        // Infinity means "unbounded"
  readonly behavior: Behavior<A>; // defined over [0, duration] in local time
}
All types are explicit, readonly where appropriate, and easy to infer in strict mode.

2. Animation clock with dt (strict & shared)
ts
import { animationFrames } from 'rxjs';
import { map, scan, share } from 'rxjs/operators';

export const animationClock: Observable<Frame> = animationFrames().pipe(
  // animationFrames gives us { elapsed, timestamp }
  map(({ elapsed }) => elapsed),
  scan<Time, Frame>(
    (prev, currentElapsed) => {
      if (prev === null) {
        return { t: currentElapsed, dt: 0 };
      }
      const dt = currentElapsed - prev.t;
      return { t: currentElapsed, dt };
    },
    null as Frame | null
  ),
  // filter out initial null
  // Strict mode: we know scan emits non-null after first step
  map((frame) => {
    if (frame === null) {
      // This should not happen due to animationFrames semantics,
      // but we guard to satisfy strictness.
      return { t: 0, dt: 0 };
    }
    return frame;
  }),
  share()
);
If you prefer, you can split the null handling out to keep the pipeline simpler.

5. Tween behavior (FRAN-like, but discrete-time)
ts
export interface TweenConfig {
  readonly from: number;
  readonly to: number;
  readonly duration: Time;
  readonly easing?: (u: number) => number; // u in [0, 1]
}

/**
 * A pure behavior that interpolates from `from` to `to`
 * over [0, duration] in local time, with optional easing.
 */
export function tween(config: TweenConfig): Behavior<number> {
  const { from, to, duration, easing } = config;
  const delta = to - from;
  const ease: (u: number) => number = easing ?? ((u) => u);

  return (t: Time): number => {
    if (t <= 0) return from;
    if (!Number.isFinite(duration) || duration <= 0) {
      // avoid division by zero, treat as jump
      return to;
    }
    if (t >= duration) return to;

    const u = t / duration; // in (0, 1)
    return from + delta * ease(u);
  };
}
No unchecked undefined, no magic branches.

6. Events and starting animations
Convert arbitrary Observables into FRAN-like events with timestamps.

import { map as rxMap } from 'rxjs/operators';

export function toEvent<A>(source$: Observable<A>): Event<A> {

7. Running an Animation<A> from a start event
ts
import { switchMap, takeWhile } from 'rxjs/operators';

export function runAnimation<A>(
  animation: Animation<A>,
  start$: Event<unknown>,
  clock: Observable<Frame> = animationClock
): BehaviorStream<A> {
  return start$.pipe(
    switchMap((startEvent) => {
      const tStart = startEvent.t;
      const { duration, behavior } = animation;

      return clock.pipe(
        map((frame): Timed<A> => {
          const tLocal = frame.t - tStart;
          return {
            t: frame.t,
            value: behavior(tLocal),
          };
        }),
        // If duration is Infinity, we never complete based on time
        takeWhile(
          ({ t }) =>
            !Number.isFinite(duration) || t - tStart <= duration,
          true // inclusive: emit last frame at t == duration
        )
      );
    })
  );
}
This respects strict types and keeps semantics clear: behavior is evaluated in local time (relative to the start event).

8. Composing animations (strict seq)
ts
export function seq<A>(
  first: Animation<A>,
  second: Animation<A>
): Animation<A> {
  const duration = first.duration + second.duration;

  return {
    duration,
    behavior: (t: Time): A => {
      if (t <= first.duration) {
        return first.behavior(t);
      }
      const t2 = t - first.duration;
      return second.behavior(t2);
    },
  };
}
Strictly typed, no mysterious branches. If you want to support heterogeneous A you can lift into a union or use Behavior<{phase: 'first'|'second'; value: ...}>, but this is the FRAN-simple version.
