RxJS Stopwatch - Plain Text Explanation
Overview
The stopwatch is a time-tracking application built entirely using reactive programming principles. Instead of using traditional imperative code with variables and if-statements, everything flows through streams of events and data transformations.

The Core Concept
Think of the entire application as water flowing through a series of pipes. User actions (button clicks, key presses) are the source of the water. The water flows through various transformation pipes (operators), and eventually comes out at the endpoints (the display, button labels, lap list) where it causes visible changes.
The key insight: nothing is directly manipulated. Instead, we describe how data should flow and what transformations should happen, and the reactive system handles all the timing and coordination automatically.

How It Works
The Event Sources (Input Streams)
The application begins by listening to user interactions. These become event streams - imagine them as rivers that emit a drop of water every time something happens:

When someone clicks the Start/Pause button, that click becomes an event in a stream
When someone presses the Space key, that keypress becomes another event
When someone clicks Reset, that's another stream of events
When someone clicks Lap, yet another stream

These streams just flow continuously, waiting for events to happen. They don't "do" anything yet - they just carry information about what the user did.
Combining Input Sources
Here's something clever: both the button clicks and keyboard shortcuts need to do the same thing. So instead of writing the logic twice, the application merges these streams together. It's like two rivers joining into one - whether the event came from a button or a keyboard doesn't matter anymore, they all flow in the same combined river.
Creating the Running State
One of the merged streams (the start/pause actions) needs to control whether the stopwatch is running or paused. This is handled by accumulating state - think of it like a toggle switch that flips every time water (an event) flows through it.
The stream starts with a value of "false" (not running). Each time an event comes through, it flips: false becomes true, true becomes false, and so on. This creates a new stream that doesn't emit clicks anymore - it emits the current running state: true, true, true... then false, false, false... then true again.
This state stream is shared among multiple parts of the application. It's like a broadcast tower - many listeners can tune in to the same signal without interfering with each other.
The Tick Stream (Conditional Behavior)
Here's where it gets really interesting. The application needs to emit "tick" events every second, but only when running. In imperative code, you'd use setInterval and clearInterval. In reactive code, you use stream switching.
The running state stream (true/false) acts as a switch. When it emits "true", the application switches to a stream that emits every second (an interval). When it emits "false", it switches to a stream that never emits anything (essentially silence).
The beautiful part: when switching happens, the previous stream is automatically cleaned up. You never have to remember to stop the interval - the reactive system handles it. This prevents memory leaks and simplifies the logic dramatically.
Building the Actions
Now we have two important streams flowing:

The tick stream (emitting every second when running)
The reset stream (emitting when someone clicks reset)

But these streams emit different things - ticks are just time pulses, resets are click events. The application transforms them into a unified concept: instead of emitting the events themselves, they emit functions that transform state.
Think of it like this: instead of saying "a tick happened", the stream says "here's a function that adds one". Instead of "reset was clicked", it says "here's a function that sets to zero".
These function-emitting streams are then merged into a single actions stream. It's a river carrying little transformation instructions.
Accumulating Time
The actions stream (carrying transformation functions) flows into an accumulator. This is like a bucket that starts at zero. Every time a function flows through, it gets applied to the current value in the bucket:

Bucket has 0, increment function arrives, bucket now has 1
Bucket has 1, increment function arrives, bucket now has 2
Bucket has 2, increment function arrives, bucket now has 3
Bucket has 3, reset function arrives, bucket now has 0
Bucket has 0, increment function arrives, bucket now has 1

This creates the time stream - a continuously updating stream of the current elapsed seconds. Every subscriber to this stream sees the latest time value whenever it changes.
Lap Times (Dependent State)
Lap times work differently. The application doesn't want to record a lap every second - only when the user clicks the Lap button. But when they do click it, the application needs to know what the current time is.
This uses a pattern called sampling. The lap click stream is the trigger, but it pulls in the latest value from the time stream only when a lap event happens. It's like taking a photograph of the time - you only capture it at the moment you click the shutter.
Each lap event, combined with the current time, gets accumulated into an array. The array grows with each lap, storing the lap number, total time, and the time since the last lap (split time).
Formatting and Display
The raw time (just a number of seconds) isn't human-friendly. A formatting function transforms it into hours:minutes:seconds with proper zero-padding. This is a pure transformation - same input always gives same output, no side effects.
Updating the Interface (Side Effects)
All the streams described so far are pure - they just transform and combine data without actually changing anything visible. The final step is subscribing to these streams and causing real-world effects:

When the time stream emits a new value, update the display element's text
When the running state stream emits, update the button label (Start ↔ Pause)
When the laps stream emits, rebuild the entire lap list in the interface

These subscriptions are the only place where the DOM actually changes. Everything else is just data flowing through pipes.
Error Handling
If something goes wrong (maybe a function throws an error), the stream doesn't just crash. Instead, it catches the error, logs it, and emits a safe fallback value (like resetting to zero). The application continues working even when unexpected things happen.
Memory Management
When the application shuts down (or in a single-page app when you navigate away), all the subscriptions need to be cleaned up. Instead of tracking them individually, they're all collected into a single container. One cleanup call stops everything - removes event listeners, cancels intervals, frees memory.

The Reducer Pattern
The core innovation here is the reducer pattern: instead of emitting values directly, streams emit transformation functions. This creates a separation between "what happened" (the event) and "how to respond" (the transformation).
Benefits:

Each transformation is a pure function that's easy to test
The state transitions are explicit and visible in the code
New behaviors can be added by just adding new transformation functions
The system is composable - small pieces combine into complex behavior


Stream Composition
The application builds complex behavior by combining simple streams:

Multiple input sources (buttons + keyboard) merge into semantic actions
Running state switches between active and inactive streams
Actions merge into a single transformation stream
Time accumulates through successive transformations
Laps sample time at specific moments

None of these streams "know about" each other directly. They just describe transformations, and the reactive system wires everything together.

Why This Approach?
Declarative vs Imperative
Traditional code says "when button clicked, if running, stop interval, else start interval". This reactive code says "running state stream switches between interval and nothing". The first describes steps, the second describes relationships.
Automatic Resource Management
No manual cleanup code. When you stop subscribing or switch streams, everything cleans itself up automatically.
Testability
Pure functions are trivial to test. Streams can be tested with marble diagrams showing exact timing. No mocking needed for most tests.
Composability
Adding a new feature means adding a new stream and connecting it to the existing flow. You don't modify existing code - you compose new behavior from existing pieces.
No Shared Mutable State
Every value flows through streams. Nothing is stored in variables that multiple parts of the code modify. This eliminates entire categories of bugs.

The Mental Model
Think of the application as a factory assembly line:

Raw materials (user events) enter at one end
They pass through various stations (operators) that transform them
Some stations combine materials from multiple sources (merge)
Some stations decide which path materials take (switchMap)
Some stations accumulate materials over time (scan)
The finished products emerge at the other end (DOM updates)

The factory keeps running continuously. Materials (events) flow through whenever they arrive. The assembly line configuration never changes - only the materials flowing through it change.
