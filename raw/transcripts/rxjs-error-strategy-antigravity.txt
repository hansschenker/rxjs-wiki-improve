In RxJS, if an error reaches the main stream, it sends an Error notification which immediately terminates the stream (unsubscribing everything). To prevent this, we catch the error inside the operator that processes each item.

Here is the step-by-step strategy used in your 
MusiciansService

Explode the Array (switchMap + from): We convert the single array [M1, M2, M3] into a stream of individual items: M1... M2... M3.
typescript
switchMap(musicians => from(musicians))
Isolate Operations (concatMap): We process each item in its own "Inner Observable" using concatMap. This creates a boundary around each item's processing logic.
typescript
concatMap(musician => { ... })
The "Inner Catch" Pattern: We attach catchError directly to the observable inside the concatMap.
If an error occurs: It is caught locally. We return EMPTY (to essentially delete the item) or a fallback value.
Crucially: The main stream sees this as a "successful completion" of that inner item, so it proceeds to process the next item in the list.
typescript
return throwError(...).pipe(
    catchError(err => {
        console.warn(err);
        return EMPTY; // "Swallows" the error so the main stream survives
    })
);
Reassemble (toArray): We collect all the surviving items back into an array to match the expected return signature Observable<Musician[]>.

You are absolutely right. This is one of the most critical patterns in RxJS for building robust applications, yet it is often misunderstood.

The key distinction is Stream Lifecycle:

Outer catchError: "If anything goes wrong, the entire process is doomed. clean up and go home." (Stream Completes/Dies)
Inner catchError: "If this specific item fails, handle it gracefully, but keep the assembly line moving." (Stream Continues)
Your implementation in 
MusiciansService
 correctly effectively demonstrates this:

switchMap (Line 58): Switches to the new data stream (handling the "latest request" logic).
concatMap (Line 59): Processes items one-by-one, ensuring order is preserved (crucial for lists).
Inner pipe + catchError (Line 63-71): Isolates the failure to the individual item. By returning EMPTY, you effectively say "ignore this item and move to the next one" rather than "crash the whole list loading process."
This pattern ensures that a single corrupted record (like your "id: 5" example) doesn't prevent the successful loading of the other 6 valid musicians.
