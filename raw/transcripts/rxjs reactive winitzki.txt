what is functional reactive programming
Based on the video, Functional Reactive Programming (FRP) is a paradigm that combines functional programming with the concept of data streams (0:53-1:39). It focuses on computing values using pure functions—meaning they have no side effects—based on immutable data (1:00-2:11).

Here are the key concepts explained in the video:

Streams: Data types that represent continuous sequences of values over time (1:14-1:40). Even though these streams conceptually update, the values within them are immutable from the programmer's perspective (1:45-2:45).
Unidirectional Data Flow: The architecture typically involves a model, view, and input stream, where pure functions compute the model from input, the view from the model, and the input from the view (2:49-5:00).
No Mutable State: Programmers do not use mutable variables. Instead, they define streams as functions of other streams, allowing the runtime to handle efficiency and updates automatically (6:04-8:26).
Primitives: FRP uses specialized primitives like map to transform streams, and scan to accumulate values over time, similar to a fold function (9:41-11:10).
Industry Application: While pure FRP is rare, libraries like React.js, Redux, and Flyd allow developers to approximate this architecture in JavaScript (14:03-15:00).

what is the difference between functional programming and functional reactive programming
Based on the video, Functional Reactive Programming (FRP) is a specialized subset of Functional Programming (FP). Here are the key differences:

Core Concepts: Functional Programming focuses on using pure functions (no side effects) and immutable data to compute values (0:53-1:00). FRP applies these same principles but adds the concept of data streams as the primary data type (1:14-1:40).
Handling Time: In standard FP, functions compute a single output from an input. In FRP, functions operate on infinite streams of values that change over time, treating time as a first-class component of the programming model (1:45-2:45).
State Management: FP avoids mutable state entirely. FRP manages changing state not by updating variables, but by defining new streams as pure functions of existing streams (6:04-8:26).

How is the model built in this system?
Based on the video, the model is built by defining a stream of model values using the scan primitive (11:45). This process involves:

Initial State: Defining an initial model value, such as a set of initial lines of code (25:32) or the starting state of a Minesweeper board (37:50).
Updating Function: Creating a pure function (the updater) that takes the old model value and an input value (like a user click) to calculate a new, immutable model value (11:49, 36:52).
Stream Accumulation: The scan function accumulates these new model values over time based on the input stream, effectively managing the application's state without direct mutation (12:02, 36:37).

model - view - input - model
This diagram describes the unidirectional data flow architecture in Functional Reactive Programming (FRP), which ensures a predictable update loop for applications (4:45-5:00).

Here is how that loop functions:

Model to View (4:15-4:18): A pure function takes the current model (state) and calculates the view (what the user sees on screen). React.js is used here to efficiently update only the changed parts of the user interface (8:03-8:26).
View to Input (3:45-4:05): User interactions within the view (like button clicks) generate inputs.
Input to Model (4:23-4:25): A pure function takes these inputs along with the previous model state to produce a new, immutable model value (11:45-12:12).
This continuous loop allows developers to build applications without managing mutable state directly (6:04-6:30).

what is the runtime responsibility in this model
Based on the video, while the programmer defines the application using pure, immutable functions and streams, the runtime system handles the practical execution and optimization (2:21-2:46):

Input Collection: The runtime actively collects inputs from the user and makes them into elements of the input stream (5:35-5:42).

How is the model updated?

Why use scan for state updates?
According to the video, the scan primitive is used to update the model because it acts like a fold function, but with a crucial difference: instead of only producing a final result, it generates a stream of intermediate values (10:40-10:55).

This is essential for building a user interface because it allows the program to accumulate a new model state based on a previous model value and a new user input event continuously (10:49-11:05).

What makes code order matter?
According to the video, in a true pure functional reactive programming (FRP) architecture using the scan primitive, the order of code initialization does not matter (31:36-32:15).

Here is why:

Conceptual Single Run: You define the streams as mutually recursive, immutable values. Conceptually, the application runs only once to set up these definitions, rather than executing procedural steps in a specific order (31:30-31:35).
Immutability: Because there are no mutable variables, functions compute values based solely on inputs, not on when they are called relative to other functions (31:45-31:55).
However, the speaker notes a limitation when using JavaScript: because JavaScript cannot handle recursive definitions of values, the architecture has to be implemented using specific libraries (Flyd) and patterns (like Redux) to simulate this behavior, which requires specific wiring of components (16:04-16:55).
