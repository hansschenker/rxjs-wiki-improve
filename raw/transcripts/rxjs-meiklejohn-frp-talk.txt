This video provides an introduction to Functional Reactive Programming (FRP) (0:07) in JavaScript. The speaker, Chris Meiklejohn, discusses the motivations behind FRP, contrasting it with traditional imperative programming by highlighting the difficulties of managing mutable state and expressing relationships over time (1:08-3:41).

Key Concepts of FRP:

Reactive Programming (2:57): Focuses on expressing how values relate over time, where changes in one value automatically trigger recomputation in dependent values (3:46). This is often seen in data flow languages like Verilog (4:07).
Functional Reactive Programming (5:17): Applies functional programming techniques (like map, filter, lift) to the reactive programming model (5:20-5:37).
Event Streams (5:47): Represent discrete occurrences of events over time, such as mouse clicks or key presses (5:57-6:31). These streams are composable using operations like filter (6:47), map (7:27), and merge (7:52).
Behaviors (Signals) (8:26): Represent values that have a continuous value over time, such as mouse position or the value of an input field (8:34-9:08). Behaviors can be transformed using functions like lift (9:11).
Switching (10:36): A mechanism to change the system's behavior based on higher-order event streams, allowing the system to observe different streams under varying conditions (10:36-11:23).
Glitch-Free Propagation (11:32): A desirable property in some FRP systems that ensures topological evaluation of dependencies to prevent inconsistent states or errors like "divide by zero" (11:32-13:10).
The video then explores several implementations and frameworks:

Flapjax (14:08): One of the first major JavaScript implementations of FRP, known for its "transparent reactivity" where operations on behaviors automatically expose and evaluate underlying event streams (15:50-18:23).
RxJS (18:27): Originating from Microsoft, this library provides reactivity through observables and observers, with features like throttle and distinctUntilChanged (18:57-21:10). It does not, however, guarantee glitch-free propagation (19:16).
Bacon.js (21:13): Inspired by RxJS and Reactive Banana, it offers similar stream manipulation and behavior changes (21:35-25:27), but also lacks glitch-free propagation (21:42).
Finally, the speaker discusses the application of FRP in Node.js (25:38) using libraries like RxJS (25:43) and how Flapjax and Bacon.js can be used in Node.js (27:10). He also mentions other options like ClojureScript libraries Shafty and Javelin (28:01), and the language Elm (29:22) for building FRP applications.

Glitch-free propagation is a desirable property in some Functional Reactive Programming (FRP) systems (11:32-11:40).

It ensures that when values propagate through a dependency graph, the system performs a topological evaluation (12:00-12:01). This means that all dependencies of an object are evaluated before the object itself is recomputed (12:04-12:11).

The video illustrates a "violation of a data flow invariant" (12:14-12:18) where, without glitch-free propagation, a depth-first evaluation could lead to errors like a divide by zero (12:46). To prevent this, the system must ensure that all dependencies are computed in the correct order (12:48-12:52).

While beneficial for correctness, implementing glitch-free propagation can slow down propagation due to the overhead of topological sorting (13:01-13:04). Not all FRP systems have this property (12:54-12:55), and its necessity depends on the application's concerns (12:59-13:02).
