export function expand<T, O extends ObservableInput<unknown>>(
  project: (value: T, index: number) => O,
  concurrent = Infinity
): OperatorFunction<T, ObservedValueOf<O>> {
  concurrent = (concurrent || 0) < 1 ? Infinity : concurrent;
  return (source) =>
    new Observable((subscriber) =>
      mergeInternals(
        source,
        subscriber,
        project as (value: T, index: number) => ObservableInput<ObservedValueOf<O>>,
        concurrent,
        undefined,
        true // Use expand path
      )
    );
}

-------
How expand works (plain explanation)

Purpose

expand takes each value from the source Observable, emits it to the subscriber, then applies a projection function (project) to that value to produce an inner Observable. Every value emitted by those inner Observables is treated like another "source" value: it is emitted and also passed to project again. This recurses until no new values are produced (or you stop it, e.g. with take).
Implementation overview

expand simply delegates the heavy lifting to the shared mergeInternals(...) routine with expand = true.
mergeInternals implements a general merging strategy used by mergeMap/mergeAll/etc., and when expand = true it switches behavior slightly so that:
every value that is being subscribed (outer or derived) is emitted immediately (destination.next(value)) before subscribing to its projected inner Observable,
every inner emission is routed back into the same "outer" processing path (outerNext), so it is emitted and projected again — that's the recursion.
Step-by-step runtime behavior

subscribe to the source observable.
for each source value, outerNext(value) runs:
if active < concurrent, doInnerSub(value) starts; otherwise the value is buffered.
doInnerSub(value) (the core):
if expand is true, immediately emit the value: destination.next(value).
increment active (count of active inner subscriptions).
call project(value, index++) and subscribe to the returned inner Observable.
when an inner Observable emits a value (innerValue):
if expand is true, call outerNext(innerValue). That will:
emit innerValue immediately (because doInnerSub emits before subscribing),
then attempt to subscribe to project(innerValue, nextIndex) (if concurrency allows) or buffer it.
if expand were false (other operators), innerValue would simply be forwarded to destination.next(innerValue).
when an inner completes:
the implementation decrements active (in finalize/complete handling) and starts any buffered values while active < concurrent.
checkComplete() is called: if the outer source has completed, buffer is empty, and there are no active inner subscriptions, then destination.complete() is called.
errors from project or inner Observables are forwarded to destination.error(...) and the subscription terminates.
Concurrency and buffering

The concurrent parameter limits how many inner subscriptions can be active at once.
If active >= concurrent, new values are pushed into a buffer and will be consumed when active drops.
expand.ts normalizes concurrency with:
concurrent = (concurrent || 0) < 1 ? Infinity : concurrent; meaning 0 or negative values are treated as Infinity. To constrain concurrency, pass a positive integer >= 1.
With concurrent = 1, expansion behaves like a sequential (depth-like) recursion: each inner emission is processed one at a time. With concurrent > 1, multiple recursive branches can be processed in parallel.
Indexing

An index counter is incremented and passed to project for each projection call (project(value, index++)), so you can know how many projections have been invoked.
Emission order / recursion character

Because doInnerSub emits a value before subscribing to its projection and because inner emissions are fed back into outerNext immediately, the pattern of processing depends on concurrency:
concurrent = 1: you get a serial chain (each produced value is emitted and fully processed before the next).
concurrent > 1: multiple inner subscriptions can interleave; the traversal is effectively concurrent and interleaved.
There is no special scheduler passed from expand(...) in this implementation (mergeInternals historically had an innerSubScheduler param but it is not used here), so scheduling is driven by the inner Observables themselves.
Practical considerations / cautions

expand can easily produce an infinite stream (e.g., project keeps returning an Observable that emits new values). Always consider limiting recursion with operators like take, takeWhile, filter, or by using a project function that eventually returns empty/complete.
If you need a specific traversal order, tune the concurrency argument.
Because every projected value is emitted and then projected again, you will see the original source values plus all recursively generated values in the resulting output Observable.
Small example (conceptual)

Suppose source emits 1, project(x) returns of(x + 1). With no limits, expand will emit: 1, 2, 3, 4, ... forever. Use take(5) after expand to get first five values.
Summary

expand works by reusing the general merge logic (mergeInternals) with expand = true.
It emits each value immediately, projects it to an inner Observable, and feeds every inner emission back into the same pipeline, producing recursive expansion.
Concurrency controls how many inner subscriptions are active; buffering ensures the concurrency limit is respected.
Be careful: expansion can be infinite unless you explicitly stop it.

Key points for concurrency = 1

Exactly one inner subscription is active at a time.
When inside an active inner subscription, any new values (outer or inner emissions) are buffered instead of immediately causing a new inner subscription.
When the active inner completes, the next buffered value is processed (subscribed) and will emit immediately (because doInnerSub emits the value to destination before subscribing to its project).
The operator therefore produces a serialized, breadth/level-like processing: each value is emitted immediately when its subscription starts, but projection (and processing of values produced by projection) happens one at a time.
With synchronous inners, the result looks like a continuous chain (no visible buffering). With asynchronous inners, the buffering produces visible serialized spacing between emitted values.

Typical usage patterns to prevent infinite emission

Use take (or takeWhile / takeUntil) after expand to stop the resulting (potentially infinite) stream.
Important: place take after expand. Putting take before expand only limits the source emissions, not the recursive emissions produced by expand.
Examples

Simple numeric chain — stop after 5 emissions
ts
import { of } from 'rxjs';
import { expand, take } from 'rxjs/operators';

const source$ = of(1);

const project = (x: number) => of(x + 1);

source$.pipe(
  expand(project),
  take(5) // stop after 5 total values (1,2,3,4,5)
).subscribe({
  next: v => console.log(v),
  complete: () => console.log('done')
});

// Console:
// 1
// 2
// 3
// 4
// 5
// done
Asynchronous expansion with delay; limit results with take
ts
import { of } from 'rxjs';
import { expand, delay, take } from 'rxjs/operators';

const source$ = of(1);

const project = (x: number) => of(x + 1).pipe(delay(100)); // async step

source$.pipe(
  expand(project),
  take(4) // prevents infinite recursion; will unsubscribe when 4 values emitted
).subscribe(v => console.log(v));

// emits: 1, 2, 3, 4 (each ~100ms apart after the first)
Limit by depth using index in project (alternative to take)
ts
import { of, EMPTY } from 'rxjs';
import { expand } from 'rxjs/operators';

const maxDepth = 3;
const project = (value: number, index: number) =>
  index >= maxDepth ? EMPTY : of(value + 1);

of(1).pipe(
  expand(project)
).subscribe({
  next: v => console.log(v),
  complete: () => console.log('complete')
});

// emits: 1,2,3,4 then completes because project returns EMPTY at index >= 3
Combine a condition and takeWhile (stop when a condition is met)
ts
import { of } from 'rxjs';
import { expand, takeWhile } from 'rxjs/operators';

of(2).pipe(
  expand(x => of(x - 1)),     // produce next value x-1
  takeWhile(x => x >= 0, true) // inclusive: emit 0 then complete
).subscribe(console.log);

// emits: 2,1,0 then completes
Notes / caveats

take unsubscribes from the expanded stream when the requested number of emissions is reached; that unsubscription will tear down active inner subscriptions, so expand won’t keep producing values after take completes.
If you place take before expand (e.g., source$.pipe(take(1), expand(...))), you only limit the number of source emissions, not the recursive emissions generated by expand.
You can also combine concurrency control (the second argument to expand) with take to control parallelism and total output size.
expand can easily produce infinite streams; always ensure you have an explicit stopping condition (take, takeWhile, takeUntil, project logic that returns EMPTY, etc.) when you expect a finite result.
