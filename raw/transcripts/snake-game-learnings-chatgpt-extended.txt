2. Observable as Functor & Monad (01:27–02:51) — Why RxJS composition works at all

This section answers a very practical question:

Why am I allowed to compose observables the same way I compose normal functions?

Markel’s answer is:
because Observable already belongs to the same family as Array and Promise.

Observable = value + context

He introduces a key abstraction:

A functor is a value with a context.

Examples he implicitly relies on:

Array<T> → context = many values

Promise<T> → context = value in the future

Observable<T> → context = value(s) over time

The important bit:

You never touch T directly

You use map to transform inside the context

So when you write:

source$.pipe(map(x => x + 1))


You are not:

Pulling a value out

Mutating it

Pushing it back

You are:

Rewriting the description of how future values should flow

That’s the functor guarantee.

Why this matters for RxJS

Because Observable is a functor:

You can:

Rename values (map)

Project values

Transform meaning

Without:

Knowing when values arrive

Knowing how many arrive

Knowing if they arrive at all

This is exactly what you want when dealing with:

User input

Game ticks

Network responses

Animations

You stay agnostic to time.

From Functor → Monad (the flattening moment)

Markel then introduces the next step:

A monad is a functor that can flatten.

In RxJS terms:

map gives you Observable<Observable<T>>

Flattening operators turn that into Observable<T>

This is where operators like:

mergeMap

switchMap

concatMap

exhaustMap

come from.

Crucially, he doesn’t present these as “async helpers”
but as composition rules.

Each one answers:

What should happen when a new inner stream appears?

This framing is essential for games.

Observable’s “context” is uncertainty

Markel highlights something subtle but powerful:

The observable context is not just “future” — it’s unknown behavior over time.

That includes:

Value may arrive now, later, or never

Value may change

Value may stop changing

Value may error

So when you compose observables, you are composing uncertainty safely.

This is why:

No mutable state is required

No “if already loaded” flags

No manual synchronization

The operators encode the rules.

Why this unlocks declarative apps

Because Observable is a monad:

You can describe your app as:

“If this changes → do that”

“When this emits → derive that”

Instead of:

“Set state here”

“Remember to clean up there”

This is what Markel means when he says:

“I can express my application as a mathematical combination of observables.”

Not math for math’s sake — but predictable composition.

Direct mapping to the Snake game

direction$ → Derived observable

snake$ → Accumulated observable via scan

Each step:

Preserves context

Never escapes the stream

Never mutates shared state

Without the functor/monad property, this would collapse into callbacks.

Markel doesn’t say it explicitly, but the implication is strong:

If something is a monad, you don’t need a framework to orchestrate it.

You only need:

Streams

Operators

One final subscribe

3. RxJS “pipe” practice (02:51–04:20) — Moving logic out of subscribe

This section is not about syntax.
It’s about where your program actually lives.

Markel is answering this question:

If observables are my program, where should the logic go?

The “non-FRP” baseline he criticizes

He starts with a very common pattern:

service.getData().subscribe(data => {
  if (data.ready) {
    const result = transform(data);
    this.state = result;
  }
});


Why this is problematic (in FRP terms):

subscribe contains:

Control flow (if)

Transformation (transform)

State mutation (this.state = …)

Logic is:

Temporal (runs now)

Implicit (hidden in callbacks)

Hard to reuse or test

In RxJS language:

The dataflow graph is invisible.

What pipe actually changes

He then shows the same logic written as:

service.getData().pipe(
  filter(data => data.ready),
  map(transform)
).subscribe(result => {
  this.state = result;
});


What changed conceptually:

pipe(...):

Describes how values will flow

Rewrites the stream

subscribe(...):

Becomes a terminal effect

No longer decides what happens — only where it lands

This is the core FRP move:

Describe first. Execute last.

Observable as a plan, not a process

Markel implicitly treats an observable chain as:

A declarative plan

A dataflow graph

A recipe

Nothing runs yet.

Only when subscribe happens does:

Time start flowing

Values start moving

Side effects begin

This separation mirrors:

SQL vs query execution

Build graph vs running the build

Iterator definition vs iteration

And it’s essential for reasoning.

Why this matters more than “clean code”

This is not about style — it’s about debuggability and composition.

When logic lives in pipe:

You can:

Reuse it

Extract it

Name it

Test it

Visualize it (marbles, graphs)

You can’t do that with logic hidden inside subscribe

Markel hints at this when he talks about clear dataflow.

Pipeable operators & tree-shaking (secondary point)

He briefly mentions why RxJS moved to pipeable operators:

Old style:

source.map(...).filter(...)


New style:

source.pipe(map(...), filter(...))


Reasons:

Better tree-shaking

Smaller bundles

Explicit imports

But for him, the real win is this:

Operators become first-class functions.

You can do:

const onlyReady = filter(d => d.ready);
const toViewModel = map(transform);

source.pipe(onlyReady, toViewModel);


Now your business rules have names.

This is where FRP becomes scalable

Once operators are named and extracted:

Pipelines read like sentences

Streams become documentation

Complexity is split across small, testable pieces

This is what makes the later Snake game readable:

direction$

snake$

score$

apples$

Each is a sentence, not a callback soup.

Subtle but critical rule

Markel’s implied rule here is:

If you feel tempted to write logic in subscribe, stop.

Ask instead:

“Can this be a transformation?”

“Is this state actually derived?”

“Should this be scan, not assignment?”

In other words:

subscribe is where effects happen, not decisions.

Why this enables games, not just apps

Games are:

Highly temporal

Continuous

Event-heavy

If your logic lives in subscribe:

You lose control of time

You lose composability

You get race conditions

By pushing everything into pipe, you:

Make time explicit

Make ordering explicit

Make policies explicit

Which is why the rest of the talk can stay pure RxJS.

});


This is exactly what FRP tries to eliminate:

External memory

Temporal coupling

Hidden control flow

It is like reduce

Not only at completion

“Given what I knew before, and what just happened, what should I know now?”

prev is memory

next is intent

The function encodes a rule

There is:

No assignment

No mutation

No side effects

Yet the rule is fully enforced.

Why this is stronger than filter

A common mistake is to think:

“Why not just filter out opposite directions?”

Because filter:

Removes events

Does not replace them

But Snake needs:

A value every tick

Even when input is invalid

scan can:

Reject the new value

While re-emitting the previous value

This is the critical difference.

So the stream stays total.

The deep insight: rules are functions, not flags

Markel implicitly teaches this:

Game rules are pure functions over time, not checks around assignments.

With scan:

Every rule is:

Local

Explicit

No global flags

No “if I remember correctly…”

instead of driver(sink$) → source$ would source(sink$) -> source$ also make sense

It can make sense, but only in a very specific way — and it’s easy to confuse people if you use source(sink$) -> source$ as your general definition of “Source”.

Why it does make sense (conceptually)

A Cycle.js Driver is really a boundary adapter:

it consumes a stream of commands (effects)

it produces a stream of observations (inputs)

So as a function shape, this is true:

boundaryAdapter(sink$) -> source$


If you rename “Driver” to “Source”, the type shape still holds:

sourceAdapter(sink$) -> source$


So yes: as long as you mean “boundary adapter”, it’s coherent.

Why it’s still risky to call that thing “Source”

In RxJS vocabulary, a source usually means:

an Observable that originates events
e.g. fromEvent(...), interval(...)

Not a function that takes sinks.

If you write source(sink$) -> source$, many RxJS folks will think:

“wait, why is a source taking input?”

“isn’t that a driver?”

So the mapping is correct, but the naming can be misleading.

A clean compromise that keeps everyone oriented

Keep Sources for the produced Observables, and name the function something like:

adapter(sink$) -> sources

connect(sinks) -> sources

createSource(...) (if it builds sources)

makeBoundary(...)

createDomSource(sinks) etc.

Even better: make it explicit that it’s a connector

Instead of returning a single source$, most boundaries return a bundle:

connectDOM(domSinks) -> domSources
connectHTTP(httpSinks) -> httpSources
connectRouter(routerSinks) -> routerSources


That reads naturally in RxJS-land: you connect effects to the outside world and get observation streams back.

Suggested naming scheme for rxjs-ui-core
Terminology

Source = an Observable you consume inside the app (click$, location$, tick$, response$)

Sink = an Observable you produce as an effect (render$, request$, navigate$)

Boundary function

Use one of these shapes:

Option A (single boundary):

createDomBoundary(...) : (domSinks) -> domSources


Option B (app runner):

run(main, boundaries)  // Cycle-like, but you don’t call them Drivers


Option C (connectors):

dom = connectDOM({ patch$ })
http = connectHTTP({ request$ })
router = connectRouter({ navigate$ })

When source(sink$) -> source$ is a good idea

If you’re intentionally offering a Cycle-style mode (even partially), then yes, the function signature is perfect — but I’d still name it something like boundary(...) or adapter(...) so you don’t fight RxJS expectations.

Bottom line

As a function shape: ✅ source(sink$) -> source$ is logically fine.

As terminology: ⚠️ calling that function a “Source” will confuse RxJS folks.

Best practice: keep Source as “input observable”, and name the function connect/boundary/adapter.

interval(ms)

Semantics:

A logical clock

Fixed-step progression

Independent of rendering

Continues even when the tab is throttled

Best for:

Simulations

Turn-based logic

Physics with fixed timestep

Game rules that must be deterministic

In FRP terms:

interval = simulation time

animationFrame

Semantics:

A render clock

Tied to the browser paint cycle

Variable cadence (≈60fps, but not guaranteed)

Pauses when tab is hidden

Best for:

Rendering

Animations

Visual interpolation

Smooth UI updates

In FRP terms:

animationFrame = display time

It keeps the example simple

Snake movement is grid-based

Visual smoothness is irrelevant

Determinism matters more than fps

For teaching FRP:

interval makes time explicit

Easier to reason about

Easier to test

So pedagogically: ✅ correct choice.

Why animationFrame is better for real UI / games

In real applications:

Rendering faster than logic is wasteful

Rendering slower than logic is janky

You want rendering to follow the screen

So you usually want:

frame$ = animationFrames();


Then decide what it clocks.

The correct FRP architecture (important)

The right move is not:

“Replace interval with animationFrame”

The right move is:

Separate simulation time from render time

Two clocks, two responsibilities
simulationTick$ = interval(100);        // game logic
renderFrame$    = animationFrames();    // rendering


Then:

State updates happen on simulationTick$

Rendering samples state on renderFrame$

  withLatestFrom(intent$),
  scan(reduceState, initialState),
  shareReplay(1)
);

render$ = renderFrame$.pipe(
  withLatestFrom(state$)
);


This is textbook FRP.

When you can use animationFrame as tick$

You can safely use animationFrame as the only clock if:

Logic is cheap

Visual smoothness matters

You accept variable timestep

You don’t need deterministic progression

Typical UI cases:

Dragging

Hover effects

Animations

Scroll-linked UI

In that case:

Snake-with-smooth-movement could use this.

FRP extends function composition into time.

Yes — and that sentence is the cleanest way to say it. Let’s unpack it carefully and operationally.

“FRP extends function composition into time”
Ordinary function composition (no time)

In normal FP, you compose values:

h = f ∘ g ∘ k


k produces a value

g transforms it

f transforms it again

Everything happens now

Order is spatial (left → right), not temporal

This is how arrays, numbers, strings, and pure data work.

What changes when time enters

Once values arrive later, arrive multiple times, or change, plain composition breaks:

You can’t call f(g(x)) if x doesn’t exist yet

You can’t “re-run” composition every time something changes

You can’t reason locally if timing is implicit

FRP answers this by lifting composition onto the time axis.

The FRP move (the key lift)

Instead of composing:

value → value → value


You compose:

value-over-time → value-over-time → value-over-time


That’s it. That’s the whole move.

In RxJS terms:

Observable<A>
  → map(A → B)
  → map(B → C)
  → Observable<C>


You are not composing values.
You are composing descriptions of how values will evolve.

What “into time” really means (concretely)
1. Functions no longer run once

They run:

whenever a new value appears

in the future

potentially many times

But you still write them as pure functions.

2. Composition preserves uncertainty

Each step preserves:

“maybe later”

“maybe many”

“maybe never”

No operator is allowed to force time.

This is why:

Observables are lazy

Nothing happens until subscription

Time comes from sources, not operators

3. State becomes composition, not storage

In non-FRP:

state = f(state, event)


In FRP:

state$ = events$.pipe(
  scan((state, event) => f(state, event), initial)
)


That is literally:

function composition across time

scan is just “fold, but stretched along a timeline”.

Why this is deeper than “reacting to events”

Reactive programming alone says:

“When something happens, do something.”

FRP says:

“Describe how things evolve when something happens.”

The difference is:

callbacks vs composition

execution vs description

control flow vs dataflow

FRP keeps the algebra, just adds time as a dimension.

Mapping your Snake game sentence-by-sentence
Traditional FP	FRP (Snake)
f(x)	map(f)
reduce(xs)	scan(xs$)
loop	tick$
mutable state	remembered stream
if/else	transition function
call stack	timeline

The Snake game is nothing but:

function composition, but each function is applied every time the clock ticks.

Why RxJS is the right tool for this

RxJS gives you:

map → rename values over time

scan → accumulate values over time

withLatestFrom → sample other timelines

flattening operators → policies for overlapping time

These are not async helpers.
They are time-aware composition operators.

The sharp formulation (worth keeping)

FRP is functional programming where time is an explicit parameter, not an implicit side effect.

Or even tighter:

FRP = FP + time as data

Why this sentence matters so much

If you internalize this:

RxJS stops looking like “event handling”

UI stops looking like “state mutation”

Framework debates (Signals vs RxJS, Redux vs local state) become secondary

You start asking only one question:

What functions should apply to which values, when time advances?

That’s the whole game.
