# **Operator Taxonomy**
### A clean, orthogonal structure for the RxJS operator landscape

---

# **The Four Core Categories**

- **Functor** — unary transformations  
- **Applicative** — combining independent Behaviors  
- **Monad** — sequencing dependent Behaviors  
- **Creation** — constructing new Behaviors  

**Speaker note:**  
These four categories form the algebraic backbone of RxJS.

---

# **Functor Operators**
## Unary Transformations

- Transform the value inside the Behavior  
- Preserve the timeline  
- One Behavior in → one Behavior out  

**Examples:**  
`map`, `filter`, `take`, `skip`, `scan`

---

# **Functor Subcategories**

### **Value Transformation**
- `map`, `mapTo`, `pluck`

### **Conditional Transformation**
- `filter`, `first`, `takeWhile`

### **Structural Transformation**
- `take`, `skip`, `startWith`

### **Accumulation**
- `scan`, `reduce`

---

# **Applicative Operators**
## Combining Independent Behaviors

- Synchronize multiple timelines  
- No dependency between sources  
- Many Behaviors in → one Behavior out  

**Examples:**  
`combineLatest`, `withLatestFrom`, `zip`, `forkJoin`

---

# **Applicative Subcategories**

### **Latest-Value Sync**
- `combineLatest`, `withLatestFrom`

### **Pairwise Sync**
- `zip`, `zipWith`

### **Completion Sync**
- `forkJoin`

### **Competition**
- `race`

---

# **Monad Operators**
## Sequencing Dependent Behaviors

- Resolve nested Observables  
- Express async workflows  
- Encode concurrency semantics  

**Examples:**  
`mergeMap`, `switchMap`, `concatMap`, `exhaustMap`

---

# **Monad Subcategories**

### **Merge Semantics**
- `mergeMap`, `mergeAll`

### **Switch Semantics**
- `switchMap`, `switchAll`

### **Concat Semantics**
- `concatMap`, `concatAll`

### **Exhaust Semantics**
- `exhaustMap`, `exhaustAll`

---

# **Creation Operators**
## Constructors (Not Part of the Algebra)

- Introduce new Behaviors  
- No transformation or combination  

**Examples:**  
`of`, `from`, `interval`, `timer`, `fromEvent`

---

# **Utility & Multicasting Operators**
## Operational Semantics

### **Multicasting**
- `share`, `shareReplay`

### **Subscription Control**
- `tap`, `finalize`, `repeat`, `retry`

### **Scheduling**
- `observeOn`, `subscribeOn`

---

# **Filtering & Rate-Limiting**
## Timeline Shaping

- `debounceTime`, `throttleTime`  
- `sample`, `auditTime`  
- `buffer`, `window`

---

# **Error Handling**
## Error as a Timeline Event

- `catchError`  
- `retry`, `retryWhen`  
- `onErrorResumeNext`

---

# **The Meta-Structure**

Functor → transforms values
Applicative → combines independent timelines
Monad → sequences dependent timelines
Creation → introduces timelines
Utility → modifies operational semantics
Filtering → shapes the timeline
Error Handling → shapes the error channel

# **Concurrency Semantics**
### How RxJS monadic operators manage overlapping timelines

---

# **Why Concurrency Semantics Matter**

- Flattening operators all implement **monadic bind**  
- But they differ in **how they handle overlapping inner streams**  
- Understanding these semantics is essential for:  
  - async workflows  
  - cancellation  
  - queueing  
  - backpressure  
  - user interaction patterns

---

# **The Four Concurrency Strategies**

- **merge** — run everything  
- **switch** — cancel previous  
- **concat** — queue sequentially  
- **exhaust** — ignore new until done  

**Speaker note:**  
These are the four “verbs” of RxJS concurrency. Every flattening operator is one of these.

---

# **mergeMap**
## Run All Inner Streams

- Starts every inner Observable immediately  
- No cancellation  
- Maximum concurrency  
- Ideal for parallel work  

**Use cases:**  
- Fire-and-forget tasks  
- Parallel HTTP requests  
- High-throughput pipelines

---

# **switchMap**
## Cancel Previous, Keep Latest

- Cancels the previous inner Observable  
- Only the latest value matters  
- Perfect for UI-driven workflows  

**Use cases:**  
- Search boxes  
- Autocomplete  
- Live-updating queries  
- Rapid user input

---

# **concatMap**
## Queue Inner Streams Sequentially

- Runs one inner Observable at a time  
- Preserves order  
- No overlap  
- Backpressure-friendly  

**Use cases:**  
- Ordered writes  
- Transactional workflows  
- Rate-limited APIs  
- Controlled sequencing

---

# **exhaustMap**
## Ignore New Until Current Completes

- Starts the first inner Observable  
- Ignores all new emissions until it completes  
- Prevents re-entry  

**Use cases:**  
- Login buttons  
- Form submissions  
- Preventing double-clicks  
- Non-reentrant workflows

---

# **Concurrency Semantics Summary**

| Operator     | Behavior            | Strategy            |
|--------------|---------------------|---------------------|
| **mergeMap** | run all             | parallel            |
| **switchMap**| cancel previous     | keep latest         |
| **concatMap**| queue sequentially  | one at a time       |
| **exhaustMap**| ignore new         | lock until done     |

**Speaker note:**  
This table is the Rosetta Stone of RxJS concurrency.

---

# **Choosing the Right Strategy**

- **Need parallelism?** → `mergeMap`  
- **Need cancellation?** → `switchMap`  
- **Need ordering?** → `concatMap`  
- **Need to block re-entry?** → `exhaustMap`  

**Speaker note:**  
Once learners understand the semantics, operator choice becomes obvious.

---

# **Concurrency Semantics as a Teaching Axis**

Parallelism  → merge
Cancellation → switch
Ordering     → concat
Reentrancy   → exhaust

Policy axioms (behavioral laws)
These are the “laws of overlap”—they’re what learners actually need.

Merge policy
No cancellation: once an inner starts, it may contribute until it terminates.

Interleaving allowed: outputs may interleave arbitrarily subject to time order.

Switch policy
Latest wins: starting a new inner cancels the previous active inner.

At most one active inner at any time.

Concat policy
FIFO sequencing: inners are queued in outer emission order.

At most one active inner; next starts only after current completes.

Exhaust policy
Non-reentrancy: while an inner is active, new outer values are ignored (not queued).

At most one active inner.

You can formalize these as constraints on the set of active inner traces at time

4) Concurrency semantics: the four verbs (the Rosetta stone)
All flattening operators are the same shape—only overlap policy differs.

Verb	Operator	Active inner streams	What happens on new outer value
merge	mergeMap	many	start another (no cancellation)
switch	switchMap	≤ 1	cancel current, start new
concat	concatMap	≤ 1	enqueue, run later
exhaust	exhaustMap	≤ 1	ignore until current completes
Selection rule (teach as a decision tree):

Need parallelism? → merge

Need “latest wins”? → switch

Need ordering / backpressure friendliness? → concat

Need non-reentrancy? → exhaust

5) Operator taxonomy derived from the algebra
5.1 The canonical classification
Creation (constructors): of, from, interval, timer, fromEvent

Functor (unary): map, filter, take, skip, scan, startWith, …

Applicative (many-in sync): combineLatest, zip, withLatestFrom, forkJoin, race

Monad (flattening): mergeMap, switchMap, concatMap, exhaustMap (+ *All variants)

5.2 “Orthogonal but not core” axes (add after the algebra)
Timeline shaping: debounceTime, throttleTime, sample, buffer, window

Operational semantics: share, shareReplay, observeOn, subscribeOn, tap, finalize

Error channel: catchError, retry, retryWhen

Teach these as additional dimensions layered on top of the core algebra.

6) A teachable narrative arc (workshop-ready)
6.1 The sequence
Behavior = value in time (events + termination)

Functor: “change the value, keep the timeline”

Applicative: “align independent timelines”

Monad: “values that spawn timelines”

Concurrency: “what overlap policy do you want?”

Everything else: timeline shaping, operational semantics, error algebra

6.2 The learner’s new question
Replace: “Which operator do I use?”
With: “Am I transforming, combining, or sequencing—and what overlap policy do I need?”
