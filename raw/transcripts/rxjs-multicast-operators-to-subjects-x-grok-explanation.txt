Simplified internal implementation logic (real source is ~100 lines — this is the essence)ts

function shareReplay<T>(
  configOrBufferSize?: number | ShareReplayConfig,
  windowTime?: number,
  scheduler?: SchedulerLike
): MonoTypeOperatorFunction<T> {
  // Normalize arguments — modern signature prefers config object
  const config = normalizeConfig(configOrBufferSize, windowTime, scheduler);

  // This factory is passed to multicast — called only once when first subscriber arrives
  const subjectFactory = () => new ReplaySubject<T>(
    config.bufferSize,
    config.windowTime,
    config.scheduler
  );

  // Create the multicasting observable
  const multicasted = multicast<T>(subjectFactory)(source);

  // If refCount === true (most common case), wrap with refCount logic
  if (config.refCount) {
    return multicasted.pipe(refCount());
  }

  // Otherwise just return the multicasted observable (stays alive forever)
  return multicasted;
}

What really happens step by stepYou do source$.pipe(shareReplay(1))
Nothing happens yet (lazy)
First subscriber arrives
→ multicast calls the factory → creates one ReplaySubject(1)
→ subscribes to the real source$ (HTTP request, timer, whatever)
→ starts multicasting values to the ReplaySubject
Values flow:
source → ReplaySubject → first subscriber
(ReplaySubject also remembers the last 1 value)
Second (third, …) subscriber arrives
→ gets connected to the same ReplaySubject
→ immediately receives the last 1 value (replay) + future values
→ no additional subscription to the source happens
Last subscriber unsubscribes (and refCount: true)
→ refCount drops to 0
→ refCount() operator unsubscribes from the source
→ source stops (HTTP request cancelled if cancellable, timer stops, etc.)
New subscriber arrives later
→ refCount goes 1 → new subscription to source is made (new HTTP call!)
→ new ReplaySubject is not created — wait, actually: yes it is!
  (when refCount hits 0 the previous subject is dropped and a fresh one is created on next connect)

Important gotchas / behavior differencesrefCount: true (default in most overloads) → cold → hot → cold again when nobody listens
→ good for HTTP caching while someone is interested
refCount: false → cold → hot forever (even after last unsubscribe)
→ dangerous with long-living sources (memory leak possible)
shareReplay(1) does not replay if source never emitted anything before subscription
(unlike BehaviorSubject which has initial value)
If source errors → error is replayed to late subscribers too
(ReplaySubject caches error/complete too)
