## Introduction to the Problem of Using Subjects in Rx
- The question of whether to use Subject in Reactive Programming (Rx) is a common source of confusion among developers, with some experts, like [[Erik Meijer (computer scientist) | Erik Meijer]], advising against its use due to common misuse, but still recognizing its necessity in certain situations.
- Erik Meijer's stance on subjects is that they should not be used in place of purely functional alternatives, and he considers subjects to be the "mutable variables" of the Rx world, which are not needed in most cases.
- Subjects have two interesting properties: they implement [[IObservable]] and IObserver simultaneously, making them suitable for both output and input, and they are hot, which is an important consideration when deciding whether to use a subject.
- To determine whether to use a subject, developers should ask themselves two questions: what is the source of the notifications (external or local), and must observers share subscription side-effects or not.

## Determining Whether to Use a Subject
- The source of notifications can be either external (an existing observable or event outside of the code) or local (an observable generated from within the code), and if the source is external, a subject should not be used, while if the source is local, a subject should be used.
- If observers do not need to share subscription side-effects, a cold observable should be defined, and if they do need to share subscription side-effects, a hot observable should be defined, which can be achieved using a subject.
- There are four possible combinations of these two questions, and the correct approach to each scenario is as follows: if the source is external and a cold observable is desired, the source can be used as is or wrapped with Defer; if the source is external and a hot observable is desired, the source can be published or wrapped with Publish, PublishLast, or Replay; if the source is local and a cold observable is desired, existing Rx generator methods can be used; and if the source is local and a hot observable is desired, defining the observable as a Subject, or a variant like BehaviorSubject, AsyncSubject, or ReplaySubject, is a potential candidate.

## Scenarios for Using Subjects
- The use of subjects in Rx is only correct in two scenarios: converting a cold observable into a hot observable and generating a hot observable imperatively and statefully without an external source, but using subjects explicitly is only correct in the latter scenario.
- To generate a hot observable imperatively and statefully without a direct external source is the primary scenario where using a subject is correct, and this is typically the case when the source is local and the observable needs to be hot.

## External Sources and Their Conversion to Observables
- External sources are defined as sources of notifications that do not need to be generated by the developer, such as events on other objects, timer callbacks, or tasks, and these can easily be converted into [[IObservable]] using various From* and To* operators in Rx.
- When working with external sources, it is recommended to use Defer or Publish to change the temperature of the observable if necessary, and then use [[Language Integrated Query | LINQ]] to query the observable, rather than using subjects, as they are not the right tool for this job.

## Local and Imperative Sources
- Local and imperative sources are defined as sources where notifications are pushed toward the outside of a stateful scope, and in this case, a subject may be a candidate for generating a hot observable, but it is essential to consider the specific requirements and constraints of the problem before making a decision.

## Alternatives to Subjects
- An alternative approach is recommended, which involves defining an observable using an existing Rx generator method, such as Return, Range, Timer, Interval, Generate, or Create, and then applying Publish with RefCount to create a shared observable.
- Rx provides various generators that make it easy to create an observable from thin air, with Create being the most versatile option, allowing for a declarative approach and supporting cancellation and auto-detachment, although it returns a cold observable.
- To create a hot observable, the Publish method can be used to convert a cold observable into a hot observable, returning an IConnectableObservable that requires a call to Connect, or by calling Publish and then RefCount to get an [[IObservable]] that is shared among multiple consecutive observers, resulting in a "warm" observable.

## Using RefCount and Avoiding Closures
- The use of RefCount makes a single subscription to the underlying observable while there is at least one observer, and disposes of the subscription when there are no more observers, making a new subscription if another observer subscribes later, and it is considered bad practice to call Connect yourself in this scenario.
- The document 'To Use Subject Or Not To Use Subject?' discusses the use of RefCount and the consideration of returning a cold observable instead, allowing callers to decide whether Publish should be used, and warns against closures when using Create, which can cause unexpected behavior due to shared variables among observers.

## Thread Safety Considerations
- The text also emphasizes the thread safety guarantees made by Rx generator methods, except for Subject, Observable.Create, and ObserverBase, which allow callers to violate some of Rx's contracts for IObserver, potentially leading to concurrent notifications and threading bugs.
- According to the Rx Design Guidelines document, Section 4.2 states that notifications must be pushed serially for Rx operators to behave properly, and Section 4.1 defines the Rx grammar as OnNext* (OnCompleted | OnError)?, which is ensured by all Rx operators and generator methods, but not by Subject and ObserverBase in the case of concurrent notifications.
- The use of Synchronize method is offered by Rx to satisfy these contracts for a given subject or observer, and it is crucial to be aware of the contracts when using these APIs to avoid difficult-to-diagnose threading bugs originating from within Rx operators due to the misuse of a subject.

## Guidelines for Using Subjects
- There is only one particular set of conditions in which it is absolutely appropriate to use subjects, and this includes representing a sequence of property changes for a particular property, exposing it with AsObservable to hide its identity, such as in the case of a Reactive Property.
- A common example of using a subject is to represent a sequence of property changes, where instead of raising an event from within the property's setter, the subject's OnNext method is called with the new value, and BehaviorSubject is often used instead of Subject because it stores the latest value of the property and pushes it immediately to new observers.
- In highly reactive programs, BehaviorSubject can encapsulate the property's value, eliminating the need for a backing field, but extracting the value imperatively is not possible without a backing field, and subscribing to the subject is the natural way of extracting the current value and all subsequent changes.
- The use of subjects can also be applied to cross-cutting concerns, such as representing cross-cutting APIs in a reactive manner, like exposing a property on a logging class that returns an observable that pushes notifications whenever something is logged, and this can be useful for real-time performance analysis, reactive heuristic-based adjustments, post-mortem debugging, and real-time debugging.

## Conclusion and Guidelines for Using Subjects
- The conclusion of the text provides guidelines on when to use subjects in Rx, stating that subjects should only be used when there is no existing observable or convertible source, a hot observable is required, the scope of the observable is a type, and there is no need to define a similar event or existing event.
- According to the text, using subjects is generally discouraged, and they should only be used as a last resort when all other options are exhausted, and even then, it is crucial to consider the potential consequences and ensure that the use of subjects does not introduce bugs or concurrency issues.
- The text also provides advice on how to work with observables, including using existing Rx generator methods, converting events or tasks into observables, and applying operators like `Publish` and `RefCount` to manage the lifetime and behavior of observables.
- The guidelines for using subjects are summarized as follows: use a subject only when you don't have an observable or anything that can be converted into one, you require a hot observable, the scope of your observable is a type, and you don't need to define a similar event and no similar event already exists, and even then, be cautious and consider the potential implications of using a subject.
