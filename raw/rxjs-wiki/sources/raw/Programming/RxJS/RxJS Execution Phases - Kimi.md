## Introducción a RxJS y conceptos clave
- RxJS is a library for reactive programming using Observables, allowing users to work with asynchronous data streams, and is part of the [[ReactiveX]] family of libraries, which includes ports for Java, Python, [[.NET]], and more.
- The key concepts in RxJS include Observables, which represent a stream of values that can be emitted over time, Observers, which react to values delivered by an Observable, Subscriptions, which connect an Observer to an Observable, Operators, which enable functional programming style transformations on streams, and Subjects, which act as both an Observable and Observer.
- RxJS is commonly used in Angular applications, but also works with React, Vue, Node.js, or vanilla [[JavaScript]], and is particularly useful for handling complex async scenarios, such as real-time data, user interactions, HTTP requests, and state management.
- The library has two distinct phases: the Execution Plan phase, which involves the creation of a cold Observable and the composition of operators using the pipe method, and the Execution phase, which is triggered by a Subscription and controlled by Schedulers, at which point the actual stream starts producing values.

## Fases de ejecución en RxJS
- RxJS provides a powerful toolkit for handling streams of data over time, with a wide range of operators, including map, filter, merge, debounce, and more, and can be used to manage complex async workflows elegantly, although it may have a steep learning curve due to its extensive operator library.
- A basic example of using RxJS is creating a stream from a search input, debouncing the input, mapping the event to a value, and subscribing to the stream to log the search value, demonstrating how RxJS can be used to handle user interactions and manage async scenarios.
- The RxJS execution phases consist of two main phases: the Execution Plan phase, also known as lazy composition, and the Execution phase, which is characterized by eager side effects and is triggered by subscription.
- In the Execution Plan phase, nothing runs, and the focus is on building a blueprint of computation, where the plan is inert, like a function definition waiting to be called, and operator chaining allows for the creation of new Observables that wrap the previous ones in onion-like layers.
- The key characteristics of the Execution Plan phase include cold Observables, operator chaining, and lazy design, which allows the plan to be passed around, composed further, stored, and treated as an object with a subscribe method.
- The Execution phase is triggered when the subscribe method is called, at which point the lazy plan becomes a hot, running process, with the producer activating, schedulers determining when and where work runs, values pushing through the operator chain, and teardown logic becoming active for cleanup when unsubscribed.
- The distinction between the two phases is important, as it allows for shared execution versus independent execution, with shared execution enabled by the use of the share operator, which allows multiple subscriptions to join the same stream, and independent execution resulting in separate timers and values for each subscription.

## Suscripción y gestión de recursos
- Subscription plays a crucial role in resource management, with the Subscription object returned by the subscribe method serving as a handle to the running process itself, and calling unsubscribe triggering the teardown logic to cancel timers, close sockets, and remove event listeners.
- Schedulers also play a key role in bridging the gap between the plan and execution phases, as they are defined in the plan phase but only manifest their behavior during execution, controlling whether next notifications are synchronous or queued for later, and enabling the creation of efficient and declarative code.

## Orígenes y evolución de RxJS
- The roots of RxJS can be traced back to the [[ReactiveX | Reactive Extensions]] (ReactiveX) project, which was started by [[Erik Meijer (computer scientist) | Erik Meijer]] and his team at [[Microsoft]] around 2010, and was inspired by functional reactive programming (FRP) concepts, with RxJS evolving from the port of Rx.NET to [[JavaScript]], and influenced by other reactive libraries such as Bacon.js.
- The origins of RxJS can be traced back to Microsoft's Cloud Programmability Team, led by Erik Meijer, who created the Volta project in 2009-2010 to handle events across client/server boundaries in tier-splitting applications.
- The theoretical foundation of RxJS is based on the mathematical dual of IEnumerable/IEnumerator, which was formalized by Erik Meijer and Gavin Bierman as IObservable/IObserver, essentially inverting "pull" sequences to "push" sequences.
- The first implementation of Rx was Rx.NET, which was released in 2011 for the [[.NET]] Framework 4.0, and was later ported to other languages, including JavaScript, resulting in RxJS, which was created by Jeffrey van Gogh and Matthew Podwysocki.
- RxJS is often mistakenly referred to as Functional Reactive Programming (FRP), but it actually deals with discrete asynchronous events, or "reactive streams," and is distinct from academic FRP, which was defined by Conal Elliott and Paul Hudak and deals with continuous time-varying behaviors.
- The RxJS project has undergone significant evolution, with major rewrites in versions 2, 4, 5, 6, and 7, and has been open-sourced on CodePlex and later moved to GitHub, with the .NET Foundation taking over in 2016.
- The community has played a significant role in the expansion of Rx, with ports to other languages, including RxJava, RxPY, and RxCpp, and contributions from key individuals, such as Bart De Smet, Wes Dyer, and Danny van Velzen.
- The distinction between "Reactive Programming" and "Functional Reactive Programming" is important, as RxJS is often colloquially referred to as FRP, but technically, it is a separate concept that deals with discrete events rather than continuous time-varying behaviors.
- The RxJS library was initially designed as a discrete collection API, avoiding continuous semantics, and was later open-sourced by [[Microsoft]] in 2012, leading to its widespread adoption and the creation of various ports, including RxJava, RxPY, and RxJS, which became the core async primitive in Angular 2+.

## Fundamentos teóricos y diferencias con FRP
- The evolution of RxJS involved several major versions, including v2-v4, which were early [[JavaScript]] ports matching [[.NET]] semantics, v5, which introduced "pipeable" operators and aligned with the ES Observable proposal, v6, which simplified imports and added tree-shaking support, and v7+, which modernized internals and introduced stricter typing.
- RxJS is often described as a combination of the Iterator pattern and the Subject/[[Observer pattern]], but this description requires nuance, as RxJS/Observable is actually the mathematical dual of the Iterator pattern, with the Iterator being pull-based and the Observable being push-based.
- The core of RxJS is the Observer pattern, which provides the subscription mechanism, including subscribe, unsubscribe, next, error, and complete, while Subjects are a specific type that combine both Observable and Observer roles, but are not the root concept.
- The most accurate way to describe RxJS is as the combination of the Observer pattern, the dual of the Iterator pattern, and functional composition, which adds operators from functional programming, making RxJS mathematically coherent and providing a powerful tool for managing async complexity in JavaScript, as noted by [[Erik Meijer (computer scientist) | Erik Meijer]] and Bart De Smet, and further developed by Ben Lesh and others.

## Patrón Observer y composición funcional
- The concept of RxJS is based on the combination of three key ingredients: the Observer pattern, the dual of Iterator, and functional composition, which together provide a unified abstraction for handling various types of event streams.
- The Iterator duality, as introduced by Erik Meijer, represents a mathematical duality where Iterable<T> (Iterator) is a pull-based sequence and Observable<T> is a push-based sequence, with the latter being the exact opposite of the former, where the producer sends values to the consumer instead of the consumer asking for values.
- The [[Observer pattern]] in RxJS consists of an Observer, which is the consumer with next(), error(), and complete() methods, and an Observable, which is the producer that accepts an Observer and decides when to push values, with a Subject being a special hybrid that is both an Observable and an Observer.
- The subscription mechanism in RxJS, which includes subscribe() and unsubscribe() methods, is adapted from the Gang of Four Observer pattern to handle resource cleanup and unicast vs. multicast semantics.
- Functional composition in RxJS is achieved through the use of pure functions, known as operators, which transform streams and are borrowed from functional programming, allowing for the creation of a "lazy assembly line" that enables sequential processing, terminal states, and composable transformations.
- The combination of the Observer pattern, the dual of Iterator, and functional composition in RxJS enables the creation of a unified abstraction for handling various types of event streams, including HTTP requests, WebSockets, user events, and more, with the operators remaining the same across different domains.

## Abstracción unificada para flujos de eventos
- The domain-specific sources in RxJS, such as fromEvent, fromFetch, web sockets, sensors, and animations, can be combined with universal operators like map, filter, debounceTime, and switchMap, which work identically regardless of the source, allowing for composition across domains and enabling the creation of declarative event streams.
- The Observable contract, which includes the subscribe(next, error, complete) method, acts as a universal interface that allows operators to work with any underlying source, making it possible to switch from one source to another without changing the transformation logic, and to combine different sources using the same operators.
- The concept of RxJS execution phases is centered around the idea of separating domain-specific sources from domain-agnostic operators, allowing for cross-domain composition and swappable sources without rewiring logic.
- Domain-specific sources, such as clicks, position, animation, stock, and sensor data, can be obtained from various domains, including the DOM, hardware, browser, network, and device hardware, using operators like fromEvent and from, and can be combined using operators like combineLatest.
- Domain-agnostic operators, such as throttleTime, map, filter, distinctUntilChanged, bufferTime, and scan, can be applied to these sources, regardless of their origin, to perform transformations like rate limiting, projection, filtering, deduplication, batching, and accumulation.
- The use of RxJS enables cross-domain composition, where wildly different domains can be combined using the same operators, and swappable sources, where the data source can be changed without rewiring the logic, as demonstrated by the example of switching from polling HTTP to [[WebSocket]] push.

## Filosofía y enfoque LINQ de RxJS
- The philosophy behind RxJS is inspired by [[Language Integrated Query | LINQ]], which treats everything as a queryable or streamable collection, making operations polymorphic across domains, and RxJS extends this concept to data in motion, or events, allowing for the creation of a domain-agnostic computation graph.
- The execution plan concept is crucial in RxJS, where operators build a computation graph, and only when subscribed does the domain-specific producer activate, pushing its particular shaped data through the generic transformation pipeline, making it easy to learn the operators once and apply them to various domains without context-switching the mental model.

## Modelo de secuencias temporales en RxJS
- RxJS Observables can be viewed as lazy, potentially infinite sequences of pairs of time and value, where the occurrence of the value implies a time dimension, and this perspective is validated by the lazy and potentially infinite nature of Observables, as well as the temporal nature of the values.
- The concept of RxJS Observables is closely related to event streams in functional reactive programming (FRP), signals/behaviors in continuous time FRP, and the Observable<T> as a function of time concept, with a subtle distinction that RxJS Observables are discrete events, not continuous functions of time.
- The "Sequence" part of RxJS Observables is lazy and push-based, not eager or pull-based, and time is usually implicit in the data structure unless operators add it, which is in contrast to Array and Promise data structures.
- This perspective on RxJS Observables is useful because it highlights the temporal nature of the events, explains why operators like debounceTime, throttleTime, and delay make sense, and connects to the "marble diagrams" used in RxJS documentation, which visualize the sequence of events over time.
- The user's perspective on RxJS Observables is valid and useful for understanding how RxJS works, and it is similar to how Conal Elliott defines semantic functions in FRP, with the Observable being a functor/monad over time-indexed values.
- The [{Time, Value}..] model captures the essence of RxJS Observables, with a lazy list of time-tagged values, where time is implicit unless materialized by operators, and this model fits because it allows for lazy evaluation, implicit vs explicit time, and infinite domains.
- The marble diagrams used in RxJS documentation are literally visualizations of the [{T,a}..] sequence, with the horizontal axis representing time and the letters representing the values, and this model reveals why marble diagrams work so intuitively.
- The concept of RxJS Observables as a sequence of time-tagged values is a sophisticated observation that connects RxJS to its theoretical FRP roots, and it is a powerful mental model for understanding how RxJS works, with time being a metavariable that is usually implicit unless materialized by operators.

## Transformaciones temporales y operadores
- The concept of Observables can be explained through the lens of temporal transformations, where operators such as map, delay, debounce, bufferTime, and zip can be viewed as modifying the temporal coordinate of the observable, with each operator having a specific effect on the time-value pairs.
- The key distinctions between Arrays, Promises, and Observables lie in their indexing, finiteness, and push/pull nature, with Arrays being space/memory indexed and finite, Promises being single time-value pairs, and Observables being time-indexed and potentially infinite sequences of pairs.
- The model of Observables as sequences of time-value pairs also highlights the difference between Events, which are discrete occurrences at times, and Behaviors, which are continuous functions of time, with RxJS lacking true Behaviors but approximating them through BehaviorSubject.
- The monadic nature of Observables is evident in the return and bind operations, which lift single values into the temporal stream and spawn new substreams at specific times, respectively, with switchMap canceling previous timelines by replacing the future sequence with a new one.

## Semántica denotacional y comportamiento temporal
- The denotational semantics of RxJS, represented by the sequence of time-value pairs, explains various aspects of RxJS, including why time-based operators feel natural, why cancellation works, and why cold Observables restart from T=0.
- RxJS Subjects can be characterized as a proxy between the Observable and Observer, enabling multicast behavior, but a more accurate description would be a hybrid or bridge that combines the properties of both Observables and Observers, with different types of Subjects, such as BehaviorSubject, ReplaySubject, AsyncSubject, and basic Subject, each with unique properties and use cases.

## Subjects y comportamiento multicast
- The Subject's dual nature as both an Observable and an Observer allows it to maintain a list of observers and push received values to all subscribers, making it a key component in enabling multicast behavior in RxJS, and its internal registry of observers and various variants make it a powerful tool for managing multiple subscriptions and executions.
- The RxJS Subject is a crucial component that acts as a proxy or bridge between a single source and multiple consumers, enabling the transition from unicast to multicast semantics, and it is both an Observable and an Observer simultaneously, implementing both interfaces.
- The Subject maintains an internal list of observers, and when the next() function is called, it iterates through the observers and calls their next() function, effectively broadcasting the data to all subscribers, which is a key aspect of the multicast mechanism.
- In contrast to unicast, where each subscriber has its own separate execution, the Subject enables multicast, where multiple subscribers receive the same data from a single execution, and this is achieved through the Subject's ability to subscribe to the source as a single Observer and then broadcast to multiple subscribers.
- The Subject's implementation is based on the [[Observer pattern]] from the Gang of Four, where it maintains a list of dependents and notifies them of state changes, and it extends the Observable class and implements the Observer interface via the next(), error(), and complete() functions.
- There are different variants of the Subject, including BehaviorSubject, ReplaySubject, AsyncSubject, and WebSocketSubject, each with its own specific multicast behavior, such as replaying the latest value or caching recent history, and these variants are used in different scenarios, such as event buses, live streams, and bidirectional messaging.
- The critical distinction between a Subject and a source is that a Subject is not just a transparent proxy, but it is the source for downstream observers, and it intercepts the data flow, hiding the complexity of multiple consumers from the source and hiding the single source from consumers, which is a key aspect of the proxy mechanism.
- The use of the term "Proxy" to describe the Subject is accurate, as it acts as an intermediary between the source and multiple consumers, and it can also be thought of as an "Interceptor" or "Bridge", but the term "Proxy" works if we think of it as proxying from the source to multiple consumers, and this mental model is helpful in understanding the Subject's role in RxJS.
- The Subject in RxJS acts as a proxy, mediating between the producer and multiple consumers, enabling the transformation of private data pipelines into shared, hot streams, which facilitates event buses, state management, and efficient resource sharing.

## Clasificación de operadores
- In RxJS, operators can be categorized into two main types: Value-based operators, which operate on the value in the {T,a} pairs, and Time-based operators, which operate on the time T or timing, with examples including map, filter, scan, delay, debounceTime, and throttleTime.
- Operators can also be classified as Lossy or Non-lossy, depending on whether they preserve all information or drop some, with Non-lossy operators guaranteeing that every input value will be represented in the output, and Lossy operators intentionally dropping values to manage throughput, memory, or timing constraints.
- The distinction between Lossy and Non-lossy operators is crucial, as it reveals the trade-offs between completeness and resource management that govern reactive stream processing, with Lossy operators often being used for performance and resource management, and Non-lossy operators being used for completeness.
- Time-based operators are often Lossy because they deal with sampling or rate-limiting continuous streams, while value-based operators can be either Lossy or Non-lossy, depending on whether they filter or transform the values, with examples including debounceTime, throttleTime, and bufferTime.
- The classification of operators into Lossy and Non-lossy is not a matter of one being better than the other, but rather a deliberate design choice, with Lossy operators being used intentionally for backpressure or sampling, and Non-lossy operators being used for completeness and accuracy.
- The RxJS library provides various operators that can be categorized into value-based and time-based operators, with each type having non-lossy and lossy operators that either preserve or drop values, respectively.
- Value-based non-lossy operators, such as `map`, `scan`, `pluck`, `pairwise`, and `materialize`, operate on the payload and ignore time, transforming the input values without losing any data, whereas value-based lossy operators like `filter`, `distinctUntilChanged`, `take`, `skip`, `elementAt`, and `single` drop values based on certain conditions or selection criteria.

## Operadores no perdidos y perdidos
- Time-based non-lossy operators, including `delay`, `bufferTime`, `windowTime`, `timestamp`, and `timeInterval`, manipulate the timing or batching of values without dropping any data, while time-based lossy operators like `debounceTime`, `throttleTime`, `sampleTime`, and `auditTime` introduce loss to handle backpressure or sampling, dropping values to manage the rate of data processing.
- The critical trade-off matrix highlights the different operator types, including value-based and time-based operators, and their respective non-lossy and lossy categories, with use case rationales for each, such as lossy operators being used for subset selection or rate limiting.
- Lossy time-based operators exist to provide intentional backpressure and handle the continuous and infinite nature of the temporal dimension, with examples including debouncing to avoid intermediate states, throttling to respect UI render cycles, and sampling to respect network bandwidth.
- The composability of lossiness in RxJS allows developers to chain lossy and non-lossy operators to create sophisticated policies, such as non-lossy accumulation followed by lossy temporal sampling and then non-lossy formatting, to achieve specific goals while managing data loss and processing constraints.

## Composición de operadores y gestión de pérdida
- The genius of RxJS lies in its ability to let developers compose these operators to balance the trade-offs between value-based and time-based lossiness, ensuring that the accumulated state that matters is never lost, while also protecting downstream resources from excessive data.
- Schedulers in RxJS serve as central dispatchers that transform the declarative Execution Plan into imperative reality, controlling the timing and execution context of actions, and they are the key to answering the question of when and where the work actually runs.

## Schedulers y control de ejecución
- The Scheduler interface maintains a queue of actions, also known as units of work, and decides when to execute them, with methods such as now() to get the current time and schedule() to dispatch work, and they control the realization of time-value pairs in the lazy sequence model.
- Schedulers dispatch three main categories of work, including notification delivery, operator execution, and subscription side effects, with examples such as observeOn(asyncScheduler) for notification delivery, interval(0, asyncScheduler) for operator execution, and scheduled([1, 2, 3], queueScheduler) for subscription side effects.
- There are several types of schedulers, including queueScheduler, asapScheduler, asyncScheduler, animationFrameScheduler, and VirtualTimeScheduler, each with its own dispatch target and use case, such as recursive operations, async but before next render/paint, standard async deferral, sync with browser repaint, and time-travel testing.
- The genius of the dispatch model is virtualization, which allows for the decoupling of time from clocks, enabling the use of virtual time schedulers for testing, and making the sequence deterministic, with the T coordinate becoming relative to the dispatcher's clock, not wall-clock time.
- Schedulers bridge the gap between the plan and execution phases, with the plan phase composing a lazy blueprint specifying what to do when time T arrives, and the execution phase being owned by the scheduler, which says "Now is T=1000, dispatch the next value", and without a scheduler, Observables default to synchronous greedy execution, which can cause stack overflows or timing bugs.
- The use of schedulers enables time-travel testing, with the VirtualTimeScheduler allowing for the execution of 1000 virtual milliseconds instantly, and the scheduler's now() method allowing for the virtualization of time-based logic, making it possible to swap a real-time dispatcher for a virtual one.
- The dispatcher architecture is based on the concept of a scheduler interface, which provides a way to schedule work and get the current time, and the different types of schedulers provide a way to control the execution context and timing of actions, with the queueScheduler, asapScheduler, asyncScheduler, animationFrameScheduler, and VirtualTimeScheduler being the main types of schedulers.
- The scheduler in RxJS acts as a throttle that prevents the stream from executing faster than the universe or browser event loop allows, providing temporal abstraction, context switching, concurrency control, and testability.
- Schedulers are central dispatchers that provide a bridge between the mathematical purity of the Observable type and the messy reality of JavaScript's event loop, enabling features such as virtual time and deterministic testing of time-based operators.

## Abstracción semántica y diseño orientado a dominios
- In RxJS, it is considered a best practice to replace standard operator chains with domain-specific functions that have user-friendly names, making the pipeline read like business logic rather than implementation details, which is achieved by using pipe and custom functions to create abstraction layers.
- This approach is related to Semantic Naming and Domain-Driven Design (DDD) applied to streams, allowing for the creation of readable, self-documenting pipelines that tell a story and separate concerns between business logic and implementation details.
- To achieve this, developers can create custom operators, which are just functions that return a source pipe, and use higher-order functions to extract operator chains into named functions, resulting in benefits such as improved readability, testability, and business alignment.
- The transformation from technical to domain-specific pipelines involves lifting RxJS from implementation plumbing to a domain-specific language (DSL), making opaque operator chains into self-documenting business logic, as demonstrated by concrete examples that show the difference between technical implementation and domain-specific code.
- The use of domain-specific functions and custom operators enables the creation of a semantic layer or domain layer, which is a key concept in RxJS that allows developers to write pipelines that are easy to understand and maintain, following the principles of Clean Code philosophy and the "lift and shift" pattern.
- The solution to managing complex RxJS execution phases is achieved through semantic abstraction, which involves creating domain narrative functions that read like requirements, such as stabilizing user input, validating queries, fetching search results, and caching recent results.
- The domain operator pattern is implemented using higher-order functions that encapsulate technical complexity, including functions like `stabilizeInput`, `validateQuery`, `fetchSearchResults`, and `cacheRecentResults`, which are designed to be reusable and readable.
- These domain functions are used to create a pipeline that can be subscribed to, such as `userSearchInput$.pipe(stabilizeInput(), validateQuery(), fetchSearchResults(), cacheRecentResults()).subscribe(displayResults)`, which provides a clear and concise way of managing the execution phases.
- The use of factory functions can create observables that read like configuration, such as `createPriceStream` and `patientMonitorStream`, which can be used to manage complex domains like financial trading and healthcare.
- The benefits of this approach include testability at the domain level, where business logic can be tested without needing to know the internals of RxJS, as well as centralized business rules, type safety via domain types, and discoverability, which allows new developers to understand the intent of the code without needing to know the technical details.
