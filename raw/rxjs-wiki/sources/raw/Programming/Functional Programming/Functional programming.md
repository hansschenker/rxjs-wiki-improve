## Introduction to Functional Programming Principles
- Functional programming is a declarative programming paradigm that models computation as the evaluation of mathematical functions, emphasizing pure functions without side effects, immutable data structures, and the avoidance of mutable state to promote composability and predictability.
- The approach of functional programming contrasts with imperative paradigms by focusing on what the program should compute rather than how to compute it step by step, enabling higher-level abstractions and mathematical reasoning about code behavior, with its roots tracing back to lambda calculus, a formal system developed by [[Alonzo Church]] in the 1930s.
- The first functional programming language, [[Lisp (programming language) | Lisp]], was introduced by [[John McCarthy (computer scientist) | John McCarthy]] in the late 1950s, inspired by lambda calculus to support symbolic computation and artificial intelligence research, and subsequent developments in the 1970s and 1980s, including languages like ML and [[Miranda (programming language) | Miranda]], advanced features such as strong typing and lazy evaluation.
- Central concepts in functional programming include higher-order functions, recursion as the principal means of control flow, immutability, lazy evaluation, and pattern matching, which facilitate referential transparency, where function calls can be replaced by their results without altering program behavior, aiding in verification and optimization.

## Core Functional Programming Concepts
- The concept of substitutability in functional programming, which is rooted in the absence of side effects and mutable state, enables equational reasoning and underpins the reliability of functional programs, allowing for optimizations such as common subexpression elimination without fear of unintended consequences.
- Referential transparency is a key property of functional programming that allows programs to be transformed algebraically for optimization or proof, and it is maintained through the use of abstractions such as monads, which encapsulate effects like I/O or state within a pure computational framework.
- In functional programming, first-class functions are treated as values with the same rights as other data types, such as integers or strings, and can be assigned to variables, passed as arguments to other functions, returned as results from functions, and stored in data structures like lists or tables.
- Higher-order functions, which accept other functions as inputs or produce functions as outputs, build upon first-class functions and facilitate abstraction and composition, with common examples including map, filter, and fold, which promote algorithmic generality and allow the same implementation to handle diverse operations by varying the supplied function.
- The technique of currying, which involves transforming a function expecting multiple arguments into a sequence of functions, each accepting a single argument, enhances expressiveness by turning multi-argument functions into chains of single-argument ones, which can then be composed or passed as higher-order arguments.
- Recursion provides the primary means of controlling program flow and implementing iteration in functional programming, by defining a function in terms of calls to itself rather than relying on mutable state or imperative loops, and allows for concise and declarative expressions of algorithms, such as computing factorials or traversing data structures.
- The use of first-class and higher-order functions, as well as recursion, fosters modularity, reduces coupling, and simplifies maintenance, while also boosting expressiveness in algorithm design, enabling concise implementations of complex logic through function composition rather than imperative loops or conditionals.

## Immutability and Data Structures
- Fixed-point combinators, such as the Y combinator, enable recursion in purely applicative settings like untyped lambda calculus, where functions lack direct self-reference, by computing a fixed point Yf = f(Yf) for any function f, and allowing anonymous recursive implementations through higher-order application alone.
- The concept of general recursion in foundational models of computation is underpinned by the construction Y = λf.(λx.f(xx))(λx.f(xx)), which is a formal representation of recursion in lambda calculus.
- In functional programming, there are two primary evaluation strategies: strict evaluation, also known as eager or call-by-value evaluation, and lazy evaluation, also known as non-strict evaluation, with strict evaluation requiring that function arguments be fully evaluated before the function is applied, as seen in languages such as [[Standard ML]] and Scheme.
- Lazy evaluation, on the other hand, defers the computation of arguments until their values are actually required during the program's execution, typically using a call-by-need mechanism to avoid redundant computations through memoization, and is the default approach in [[Haskell]], facilitating the definition and manipulation of infinite data structures.
- The choice between strict and lazy evaluation strategies has significant implications, with strict evaluation offering predictability in execution order and efficiency for finite computations, but potentially leading to unnecessary work and non-termination if an argument diverges, while lazy evaluation promotes conciseness, enhances support for higher-order functions, and enables parallelism, but may result in unpredictable memory usage and challenges in debugging.
- The Church-Rosser theorem provides a foundation for the independence of these evaluation strategies in lambda calculus, stating that if a term reduces to two different terms via beta-reduction, there exists a common term to which both can further reduce, ensuring that strict and lazy strategies will converge to the same result, justifying the use of laziness without loss of correctness in confluent systems like pure functional languages.
- Reduction strategies in functional programming define the order in which expressions are evaluated to compute results, with applicative-order reduction and normal-order reduction being two common strategies, corresponding to strict and lazy evaluation, respectively, and the choice of reduction strategy can significantly impact the behavior and performance of functional programs.

## Functional Programming vs. Imperative Paradigms
- The strategies of normal-order and applicative-order reduction serve as foundational approaches in functional programming, corresponding to lazy and strict evaluation models, respectively, and influencing efficiency, termination behavior, and resource usage.
- Normal-order reduction prioritizes the leftmost outermost redex, delaying evaluation of arguments until they are needed, which ensures that if a normal form exists, it will be reached without reducing unused subexpressions, and is particularly effective in pure functional languages for avoiding unnecessary work.
- Applicative-order reduction, on the other hand, evaluates the rightmost innermost redex first, fully reducing arguments before applying functions, which mirrors eager evaluation and can prevent non-termination in some scenarios but risks computing unused values, and is common in strict languages like Scheme.
- Call-by-need extends normal-order by introducing sharing through memoization, where arguments are evaluated at most once and stored for reuse, reducing redundancy while preserving laziness, and is formally defined in the λaf calculus, as seen in languages like [[Haskell]].
- Optimal reduction strategies, such as John Lamping's algorithm, leverage graph rewriting to perform the minimal number of beta-reductions necessary to reach the normal form, eliminating all duplication and unnecessary steps, and are applied in compilers for functional languages, such as those using SECD machines or [[G-machine]] variants.
- Functional programming languages also employ various typing disciplines, including static typing, to ensure program correctness, ranging from basic type checking to advanced mechanisms that enhance expressiveness and safety, as seen in languages like Haskell, which performs type checks at compile time and enables optimizations such as type-directed compilation.
- The concept of dynamic typing, as seen in languages like [[Lisp (programming language) | Lisp]], defers type checking to runtime, offering greater flexibility for rapid prototyping, but potentially leading to errors that are only detected during execution, whereas static typing prioritizes the prevention of type mismatches before runtime.
