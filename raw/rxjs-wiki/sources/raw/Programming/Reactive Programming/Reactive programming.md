## Introduction to Reactive Programming
- [[Reactive programming]] is a declarative programming paradigm that focuses on the propagation of changes through time-varying data streams and events, enabling automatic updates in dependent computations without explicit imperative control, and it treats data flows as first-class entities, allowing developers to define relationships between values and let the system handle asynchronous updates, propagation, and composition of these streams.
- The roots of reactive programming trace back to the 1980s with synchronous dataflow languages like SIGNAL and LUSTRE, and it was further developed in the 1990s with the emergence of functional reactive programming (FRP), pioneered by works such as Paul Hudak's Fran language in [[Haskell]], which modeled continuous behaviors and discrete events as composable signals.
- Reactive programming revolves around key concepts such as behaviors (continuous time-varying values), events (discrete streams of occurrences), and mechanisms for change propagation, often using push-based (event-driven) or pull-based (demand-driven) evaluation models, and it supports features like backpressure to manage overwhelming data flows in asynchronous environments, ensuring systems remain responsive under load.
- The benefits of reactive programming include improved scalability, resilience to failures, and elasticity in handling variable workloads, making it ideal for modern applications like web services, mobile UIs, and big data processing, and notable implementations include the [[ReactiveX | Reactive Extensions]] (Rx) library and the [[Reactive Streams]] initiative, which provides a protocol for asynchronous stream processing with non-blocking backpressure.

## Core Concepts and Principles
- Reactive programming revolves around key concepts such as behaviors (continuous time-varying values), events (discrete streams of occurrences), and mechanisms for change propagation, often using push-based (event-driven) or pull-based (demand-driven) evaluation models, and it supports features like backpressure to manage overwhelming data flows in asynchronous environments, ensuring systems remain responsive under load.
- The benefits of [[Reactive programming | reactive programming]] include improved scalability, resilience to failures, and elasticity in handling variable workloads, making it ideal for modern applications like web services, mobile UIs, and big data processing, and notable implementations include the [[ReactiveX | Reactive Extensions]] (Rx) library and the [[Reactive Streams]] initiative, which provides a protocol for asynchronous stream processing with non-blocking backpressure.
- Reactive programming is concerned with data streams and the automatic propagation of changes, particularly suited for building event-driven and interactive applications, and it models software systems as asynchronous streams of events or signals, where transformations and dependencies are expressed declaratively rather than through step-by-step instructions, enabling developers to focus on what the program should do in response to changes.
- Key principles of reactive programming include the representation of time-varying values, such as behaviors and events, and the automatic management of dependencies between them, and unidirectional data flow ensures that changes originate from a single source and propagate downstream through composable stream operations, promoting predictability and reducing side effects, as outlined in the 2014 Reactive Manifesto, which enables responsive, resilient architectures for distributed environments.
- Reactive programming is a paradigm that automates propagation through declarative subscriptions, eliminating the need for explicit loops, polling, or callbacks, and instead shifts the burden from procedural control to stream composition, making it easier to reason about complex interactions without mutable shared state.

## Dataflow Graphs and Propagation
- In dataflow graphs, nodes represent individual computations or time-varying values, and directed edges denote data dependencies between them, with propagation operating through bottom-up, push-based mechanics, where an update to a source [[Node.js | node]] automatically triggers recomputation in all dependent nodes along the connected edges.
- The core transformation in dataflow graphs can be expressed as Δoutput = f(Δinputs), where f is the function defining the node's computation, and Δ denotes changes in values propagating from inputs to output, and graph construction varies between implicit and explicit approaches, drawing from foundational work in functional [[Reactive programming | reactive programming]].
- The construction of dependencies in reactive programming can be either implicit or explicit, with implicit construction automatically tracking dependencies during program evaluation, as seen in systems like FrTime, and explicit construction requiring manual wiring of nodes and edges, as seen in domain-specific tools for visual programming.
- For clarity and debugging purposes, these graphs can be visualized as diagrams with nodes as boxes labeled by computations and arrows indicating dependency directions, and timestamps are employed to order updates and resolve potential race conditions, especially in concurrent settings.

## Degrees of Reactivity
- Reactive programming encompasses varying degrees of reactivity, ranging from fully implicit models, where the system automatically detects and tracks dependencies, to explicitly defined connections that demand manual intervention, with hybrid approaches blending elements of both to balance usability and control.
- Implicit reactivity represents the most automated end of the spectrum, where changes propagate through automatic dependency tracking without developers explicitly specifying connections between data sources and dependents, as exemplified by spreadsheets like [[Microsoft Excel]] and modern frameworks like [[Svelte]].
- Explicit reactivity, on the other hand, requires developers to manually declare dependencies, often through subscriptions or bindings, granting precise control over change propagation, but increasing code verbosity, as seen in the [[ReactiveX | Reactive Extensions]] (Rx) library and functional [[Reactive programming | reactive programming]] (FRP) systems like Fran.
- Hybrid degrees of reactivity, such as push-pull models, integrate implicit automation with explicit control to mitigate the limitations of pure forms, allowing for the ease of implicit tracking for simple cases alongside explicit tuning for efficiency, as evident in FRP libraries like NewFran.
- The choice of reactivity degree often depends on the application's scale and requirements, with trade-offs in developer effort, performance, and predictability, and each approach has its advantages and drawbacks, such as reduced boilerplate and easier onboarding for implicit reactivity, and fine-grained optimization and clearer visibility into event lifecycles for explicit reactivity.
- The concept of hybrids in reactive programming offers a middle ground between implicit and explicit models, providing a balance between ease of use and performance control, with implicit elements enhancing productivity and explicit elements preserving control.
- The evolution of reactive programming models has progressed from early implicit systems, such as spreadsheets in the 1980s, to explicit models in the 1990s, including pioneering work in Fran in 1997, which emphasized explicit signal connections for reliability in animation and simulation domains.

## Programming Paradigms in Reactive Programming
- The section on programming paradigms in the context of [[Reactive programming | reactive programming]] discusses two primary approaches: functional and imperative/procedural methods, each with its own strengths and use cases for building reactive systems.
- Functional approaches to reactive programming, such as Functional Reactive Programming (FRP), emphasize the composition of pure functions over streams of events and behaviors, avoiding side effects to ensure referential transparency and predictable execution, and leveraging higher-order functions like map, filter, and flatMap to transform and combine event streams declaratively.
- The functional reactive programming paradigm treats time as a first-class citizen, modeling continuous time-varying values as behaviors and discrete changes as events, which are streams of timestamped occurrences, and reactive programs are expressed through pure functional compositions, such as sampling a behavior at event times or integrating event occurrences into new behaviors.
- Influential variants of FRP, such as arrowized FRP, structure these compositions using arrow combinators to define signal functions that process input streams to output streams, facilitating efficient handling of both continuous dynamics and discrete transitions without explicit time representation, and libraries like Reactive-Banana provide practical implementations of the signals-and-events model.
- Imperative and procedural methods, on the other hand, integrate reactive principles into sequential, state-mutating codebases by embedding event-driven mechanisms that respond to data changes through explicit control structures like loops and assignments, and allow developers to retain familiar imperative flows while adding reactivity, often by altering read/write semantics on variables to propagate changes incrementally.
- Examples of imperative and procedural methods include wrapping imperative code in reactive callbacks or observables, using event-driven loops to process sequences of events, and employing libraries like [[ReactiveX | Reactive Extensions]] (Rx.NET) in languages like C#, which combine asynchronous programming constructs like async/await with reactive operators to manage data streams.
- The use of reactive extensions, such as those provided by Reactive-Banana and Rx.NET, offers advantages in composability and testability, as pure functional building blocks allow reactive systems to be assembled hierarchically and reused across contexts, and side-effect-free compositions can be verified unit-wise independent of runtime timing.
- The [[Reactive programming | reactive programming]] approach enables reactive behaviors in scripts that process file I/O or network requests imperatively, with subscriptions driving the flow, and this is achieved through the use of interfaces like IObservable<T> and IObserver<T> in C# procedural code.

## Object-Oriented Reactive Programming
- In object-oriented reactive programming, objects serve as the primary units of reactivity, encapsulating mutable state and enabling automatic propagation of changes through observable properties, and this approach leverages encapsulation to bundle reactive behaviors within classes, ensuring that internal state mutations notify external observers without exposing implementation details.
- Reactive objects extend traditional OOP by integrating streams of state changes into class instances, where assignments to reactive fields trigger dependency reevaluations in dependent methods or objects, and inheritance plays a key role in composing reactive behaviors, allowing subclasses to inherit reactive fields and dependencies from superclasses while maintaining propagation semantics.
- Common patterns in object-oriented reactive programming include property observers, where getters and setters in classes monitor changes to trigger updates, and frameworks like Knockout.js use observable properties created via ko.observable(), which function as reactive getters and setters, assigning a new value notifies bound UI elements for automatic synchronization.
- The use of interfaces like PropertyChangeListener in Java supports bound properties in beans, firing events on state changes to enable [[ReactiveX | reactive extensions]], and libraries like Frappé convert Java bean properties into reactive behaviors using PropertyChangeListener, allowing declarative composition of streams from object mutations.
- The benefits of object-oriented [[Reactive programming | reactive programming]] techniques include strong encapsulation of reactive state, where objects hide propagation logic behind interfaces, and polymorphism, enabling swappable reactive modules through abstract base classes, but limitations arise from tight coupling in mutable objects, where direct field assignments can create unintended dependency graphs leading to glitches.
- The concept of reactive programming involves handling cyclic dependencies in inherited reactive methods, which can amplify propagation delays or errors and complicate debugging in large class hierarchies, emphasizing the need for effective management of concurrent and distributed models.

## Concurrent and Distributed Models
- In reactive programming, concurrent and distributed models are designed to handle parallelism and network distribution by emphasizing asynchronous communication and fault tolerance, with the actor model being central to this approach, where actors are treated as the fundamental units of computation that respond to incoming stimuli, such as messages or events.
- The actor model in reactive programming enables scalable concurrency without shared mutable state, with actors interacting solely through message passing, and this approach ensures location transparency, where actors communicate identically regardless of whether they reside on the same process or across distributed nodes, facilitating seamless scaling in reactive systems.
- The [[Akka (toolkit) | Akka]] framework is an example of a [[Reactive programming | reactive programming]] framework that implements reactive actors that respond to messages in a non-blocking manner, supporting elastic distribution through remoting and clustering mechanisms that maintain system responsiveness under load, and allowing actors to form hierarchies for supervision, enabling reactive feedback where parent actors monitor and restart children upon failure.
- Rule-based reactivity is another approach that complements the actor model, defining responses through event-condition-action (ECA) rules, where events trigger condition evaluations that, if satisfied, execute actions in distributed settings, and this declarative approach suits complex event processing (CEP) in reactive systems, enabling near-real-time detection and reaction to patterns across event streams from multiple sources.
- Concurrency in reactive programming models is managed through mechanisms like backpressure, which signals producers to slow down when consumers in distributed streams are overwhelmed, preventing cascading failures, and backpressure is enforced via protocols that limit message flow, as standardized in [[Reactive Streams]], allowing systems to remain elastic.
- Distributed propagation in reactive programming involves algorithms that synchronize changes across nodes while preserving glitch-freedom and eventual consistency, often guided by principles from the Reactive Manifesto for resilient systems, and techniques like the QPROP algorithm enable asynchronous, decentralized propagation of reactive values in dependency graphs spanning multiple services.
- A key concept in reactive programming is the message-driven architecture with reactive feedback, where components exchange asynchronous messages to drive reactions, incorporating backpressure and supervision for loose coupling and fault tolerance, and this approach enables location-transparent interactions that scale across distributed clusters, with feedback loops ensuring adaptive responses to dynamic loads.

## Implementation Strategies
- [[Reactive programming]] also involves implementation strategies, including static vs. dynamic reactivity, where static reactivity refers to approaches where dependency analysis and graph construction occur at compile time, enabling optimizations such as fixed dataflow graphs that enhance performance and safety, and this paradigm allows the compiler to examine code to identify relationships between reactive values and generate efficient update mechanisms without runtime dependency discovery.
- The concept of reactive programming employs two primary approaches, static reactivity and dynamic reactivity, each with its own set of advantages and disadvantages, where static reactivity involves compile-time analysis to build a directed acyclic graph of dependencies, allowing for predictable propagation and avoiding inefficiencies from dynamic introspection, as seen in frameworks like [[Elm (programming language) | Elm]] and [[Svelte]].
- Static reactivity, as exemplified by frameworks such as [[Solid JS | SolidJS]] and Svelte, uses compile-time tracking for its proxy-free signals, where dependencies are resolved statically to enable fine-grained updates without runtime overhead, resulting in smaller bundle sizes and faster initial renders, as updates are pre-wired without ongoing tracking overhead.
- In contrast, dynamic reactivity, as implemented in [[JavaScript]] frameworks like [[React (software) | React]], builds and modifies dependency graphs at runtime, providing flexibility for scenarios involving user-driven structural changes or conditional dependencies that cannot be fully anticipated at compile time, but introduces overhead from continuous observation and reconciliation.
- The trade-offs between static and dynamic reactivity center on performance, safety, and adaptability, where static methods offer superior speed and reliability via precomputed graphs, along with compile-time error detection, but limit flexibility for highly dynamic environments, while dynamic approaches excel in adaptability, supporting runtime graph reconfiguration essential for interactive UIs, yet suffer higher execution overhead from dependency resolution.
- Hybrid systems, such as Svelte 5's runes, mitigate the trade-offs between static and dynamic reactivity by performing partial evaluation at compile time while incorporating dynamic extensions, blending static optimizations with flexible tracking where needed, enabling runtime reactivity for shared state across components.

## Change Propagation Algorithms
- Change propagation algorithms in reactive programming manage the efficient dissemination of updates through dependency graphs, ensuring that dependent computations reflect changes with minimal overhead, operating on directed acyclic graphs (DAGs) representing dataflow relationships, where nodes denote reactive entities and edges indicate dependencies.
- The fundamental approaches to reactive programming include push propagation, which immediately forwards changes from a source [[Node.js | node]] to its dependents using breadth-first search (BFS) or depth-first search (DFS), and pull propagation, which evaluates nodes on demand, pulling values from dependencies only when required, with hybrid push-pull models combining both for optimized throughput.
- The propagation cost in graph traversals is typically O(V+E), where V is the number of vertices (nodes) and E is the number of edges (dependencies), and optimization techniques such as delta propagation, which transmits only the differences between old and new values, can further enhance efficiency.
- Topological sorting preprocesses the directed acyclic graph (DAG) to linearize nodes in dependency order, enabling glitch-free traversal, and stabilizing propagators ensure convergence to a quiescent state by incrementally merging partial information and propagating only changed inputs.
- [[Reactive programming]] employs various evaluation and execution models, including synchronous models, which treat computations as occurring in lockstep with an external clock, and asynchronous models, which enable non-blocking execution and rely on schedulers and event loops to dispatch reactions.

## Evaluation and Execution Models
- Reactive programming employs various evaluation and execution models, including synchronous models, which treat computations as occurring in lockstep with an external clock, and asynchronous models, which enable non-blocking execution and rely on schedulers and event loops to dispatch reactions.
- Evaluation strategies in reactive systems also include eager and lazy approaches, with eager evaluation computing all dependent values immediately upon a change, and lazy evaluation deferring execution until a consumer requests data, reducing computational waste in scenarios with sporadic access.
- The choice of evaluation and execution model depends on the specific requirements of the application, such as the need for instant consistency, scalability, or responsiveness, with examples including desktop user interfaces, real-time control systems, and web servers, which may use frameworks like Flapjax, an functional reactive programming (FRP) language for Ajax, or libraries like [[Node.js]].
- [[Reactive programming]] is a paradigm that generalizes the observer pattern by extending one-to-many notifications from single events to continuous streams of data, allowing for the handling of time-varying data, such as user interactions or sensor inputs, in a composable manner.
- In reactive programming, lazy evaluation defers computation until a value is explicitly requested by a downstream observer, optimizing resource use in stream-based systems by avoiding premature work, and libraries like [[rxjs | RxJS]] default to lazy execution, only activating upon subscription.
- The time complexity of execution in reactive models varies by strategy, with synchronous models often exhibiting O(n propagation per cycle for n dependencies due to blocking completeness, whereas asynchronous models achieve O(1) amortized latency per event through non-blocking dispatching, assuming bounded scheduler overhead.

## Challenges and Issues in Reactive Programming
- Reactive programming is prone to glitches and temporal inconsistencies, which manifest as transient errors resulting from out-of-order updates during change propagation, where dependent computations temporarily reflect inconsistent or stale values, and can be detected using timestamping events or versioning mechanisms.
- Glitches can be addressed using stabilization techniques, such as topological sorting of the dependency graph, two-phase propagation, and mode-based switching, which restructure propagation to prevent recomputations on stale data and ensure updates occur in dependency order.
- Cyclic dependencies and feedback loops are another challenge in [[Reactive programming | reactive programming]], where components in a dependency graph mutually influence each other, forming feedback loops that can lead to repeated evaluations until stability is achieved, and can be detected using algorithms like depth-first search (DFS) during the construction or analysis of the dependency graph.
- The detection and mitigation of glitches and cyclic dependencies are crucial in reactive programming, as they can significantly impact the user experience in real-time systems, causing visual flickering in UIs or erroneous outputs that erode trust in interactive applications, and can lead to redundant computations and perceived unreliability without proper mitigation.
- In reactive systems, Depth-First Search (DFS) is applied recursively from each [[Node.js | node]] to identify cycles and prevent runtime issues, and this approach is efficient for large graphs due to its linear complexity in terms of the number of nodes and edges.
- Resolution strategies in reactive systems often involve relaxation methods, such as fixed-point iteration, where values are iteratively updated until convergence, or introducing delays to break the cycle explicitly, and fixed-point iteration proceeds by repeatedly applying a function to initial values until the change falls below a threshold.
- In synchronous reactive models, non-strict actors like delays allow partial evaluation, enabling the system to resolve loops by propagating unknown values initially and iterating until a fixed point is reached, and examples of these concepts appear in simulation models, such as a 2-bit counter where feedback loops are resolved using non-strict delays.
- Interaction with imperative state is a significant challenge in reactive programming, and developers employ patterns like reactive wrappers, immutability proxies, and atomic updates to address issues like race conditions and lost reactivity, and techniques like lenses provide a functional-style alternative for accessing and updating nested immutable structures.

## Scalability and Backpressure

- [[Reactive programming]] is essential for handling high-volume data streams asynchronously, using non-blocking techniques to signal demand and avoid unbounded resource consumption, and common strategies for managing producer-consumer rate mismatches include buffering, dropping, and throttling, which are implemented through operators like onBackpressureBuffer, onBackpressureDrop, and onBackpressureLatest.

- Scalability challenges in reactive systems often arise from unbounded queues, which can lead to memory leaks or OutOfMemoryErrors if producers outpace consumers, and backpressure is necessary to coordinate load across distributed nodes and ensure elastic resource allocation without cascading failures, particularly in microservices architectures.
- Recent advancements, such as those in [[RxJava]] 3, enhance flow control by refining Flowable types with explicit backpressure strategies, including bounded buffering and error handling for overflow, and practical implementations adjust buffer sizing based on observed rate imbalances, latency, and variability to absorb excess emissions over time.
