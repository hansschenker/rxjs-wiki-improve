## Introduction to RxJS and Core Concepts
- RxJS, or [[ReactiveX | Reactive Extensions]] for [[JavaScript]], is a library for reactive programming using Observables that makes it easier to handle asynchronous data streams, allowing for the management of events, async operations, API calls, user input, timers, WebSockets, and more in a clean, composable, and powerful way.
- The core idea of RxJS is that everything is a stream, and these streams can be transformed, combined, filtered, and reacted to in an elegant manner, with the most important building blocks being Observables, Observers, Subscriptions, Operators, and Subjects.
- Observables are the things that produce values over time, similar to a YouTube channel, and can be created using functions such as of(), from(), interval(), and fromEvent(), while Observers are the things that listen and react to these Observables, similar to a subscriber, and are defined by their next, error, and complete functions.
- Subscriptions represent the connection between Observables and Observers, similar to a YouTube subscription, and are created using the subscribe() function, while Operators are transformation functions for streams, similar to video editing tools, and include popular ones such as map, filter, mergeMap, switchMap, debounceTime, take, and catchError.
- Subjects are Observables and Observers in one object, similar to a multicast channel, and can be created using new Subject(), BehaviorSubject, and ReplaySubject, allowing for multiple subscribers to receive the same stream of values.

## The Three Pillars of RxJS
- RxJS provides a range of quick real-world examples, including handling button clicks, search inputs with debounce, and combining multiple API calls, demonstrating its ability to simplify complex async flows and coordinate multiple sources of events and data.
- The most popular Operators in RxJS include transformation Operators such as map, pluck, and mapTo, filtering Operators such as filter, distinctUntilChanged, and debounceTime, flattening and combining Operators such as mergeMap and switchMap, and error handling Operators such as catchError and retry.
- The current modern RxJS style, as of 2024-2026, involves importing specific functions from the 'rxjs' library and using the pipe() function to chain Operators together, creating a clean and readable code structure.
- RxJS is particularly useful when dealing with complex async flows, multiple sources of events or data, real-time features, and state derived from multiple streams, but may not be necessary for very simple async operations or small components with a single API call.
- RxJS has two phases: the declaration or construction phase, where the execution plan is built using Observables and Operators, and the execution phase, which is triggered by subscription and executes the plan, allowing for a clear separation of concerns and a more efficient use of resources.
- The RxJS library is based on three orthogonal pillars: Observable, Operators, and Schedulers, which are largely independent of each other and can be combined in many different ways to produce the desired outcome.

## Declaration vs Execution Phases
- The process of working with RxJS involves two main phases: the Declaration Phase, where the source Observable is created and operators are chained to transform, filter, combine, handle errors, and so on, and the Execution Phase, which begins when the subscribe method is called, triggering the execution of the planned operations.
- During the Declaration Phase, the code is only building a plan and does not execute any operations, such as making network requests, starting timers, or attaching event listeners, and the Observable is lazy by default, meaning it does not produce any values until it is subscribed to.
- The Execution Phase is activated through subscription, where the subscribe method is called, and this is when RxJS starts attaching event listeners, making HTTP requests, and pushing values through the operator chain, and the code begins to run and produce side effects.
- The key characteristics of the Declaration Phase include laziness, the ability to reuse the same Observable multiple times, and the inability to cancel the operation, whereas the Execution Phase is eager, allows for cancellation through the unsubscribe method, and creates a new execution for each subscription.

## Cold vs Hot Observables
- The separation between the Declaration Phase and the Execution Phase is the source of several important behaviors in RxJS, including the ability to prepare complex data flows without causing side effects, the difference between cold and hot Observables, and the ability to prevent memory leaks by unsubscribing from Observables.
- Cold Observables restart their work for each new subscriber, whereas hot Observables share the same execution among subscribers, and this difference has important implications for how Observables are used and shared in an application.

## History and Origins of Reactive Extensions
- The Observable is the central abstraction in RxJS, representing a push-based collection that may emit zero or more values over time, and it is the only thing that can produce values and initiate work, following the Observer pattern contract and being lazy by default.
- The RxJS library is based on three main pillars: Observables, Operators, and Schedulers, which work together to enable reactive programming in imperative languages by treating asynchronous and event-based data as composable observable sequences.
- Observables are producers of values that can be subscribed to, such as the `interval` function, which produces a sequence of numbers, or the `ajax` function, which produces a value or an error, and they are lazy, meaning they only produce values when subscribed to.
- Operators are pure functions that take an Observable and return a new Observable with modified behavior, such as `map`, `filter`, `debounceTime`, and `switchMap`, which can be used to transform, filter, and combine streams of data, and they are also lazy, meaning they only process values when subscribed to.
- Schedulers are the mechanism that controls when and where Observables produce values, and when notifications are delivered to subscribers, with built-in schedulers including `queueScheduler`, `asapScheduler`, `asyncScheduler`, `animationFrameScheduler`, and `VirtualTimeScheduler`, each with its own execution context and typical use case.
- The three pillars of RxJS are mostly decoupled, allowing developers to change one pillar without affecting the others, such as changing the producer, operators, or scheduler, and this orthogonality is a key feature of the library's design.
- The Quick Summary Table provides an overview of the three pillars, including their responsibilities, what they decide about, whether they are lazy, and their typical change frequency, with Observables and Operators being very often changed, and Schedulers being occasionally changed.
- [[ReactiveX | Reactive Extensions]] (Rx) originated at [[Microsoft]] as a library designed to enable reactive programming in imperative languages, with the concept emerging in the late 2000s within Microsoft's Cloud Programmability Team, which sought a unified way to handle asynchronous data flows, events, and sequences using a consistent, composable model.
- The key insight behind Reactive Extensions was to generalize the [[Language Integrated Query | LINQ]] model to time-based and asynchronous sources, such as events, timers, streams, and futures, which was driven by [[Erik Meijer (computer scientist) | Erik Meijer]], a prominent programming language researcher at Microsoft, along with a small core team including Wes Dyer, Jeffrey van Gogh, Bart De Smet, and Danny van Velzen.
- The work drew theoretical inspiration from functional reactive programming systems, the observer pattern, iterator pattern, monadic composition, and the duality between push-based and pull-based collections, with the principal architect, Erik Meijer, being supported by his team in developing the Reactive Extensions.
- The development of Reactive Extensions began internally at Microsoft around 2008-2009, with the first public presentation at the Microsoft Professional Developers Conference in 2009, followed by early community exposure through podcasts and Channel 9 videos, and the first official release of Rx for .NET in June 2011.

## Reactive Streams and Backpressure
- The [[ReactiveX | Reactive Extensions]] were later open-sourced by [[Microsoft]] in 2012, initially hosted on CodePlex and later migrated to GitHub, with ports to other platforms, including [[JavaScript]], [[Java (programming language) | Java]], C++, and Python, beginning to emerge around 2012-2013.
- The [[Reactive Streams]] specification emerged in the JVM ecosystem around 2013-2015, which was a complementary but separate initiative influenced by Rx ideas, and the Reactive Manifesto was published in 2014, with Rx concepts aligning closely with reactive systems manifesto.
- The original implementation of Reactive Extensions was Rx.NET, which was later ported to other languages and runtimes, becoming collectively known as ReactiveX, with RxJS being the JavaScript variant, and the unifying specification is the Observable contract and the set of composable operators.
- The core duality that underpins Reactive Extensions is the symmetry between pull-based and push-based sequences, allowing the same compositional style to apply to both worlds, which was a powerful idea explicitly pursued by [[Erik Meijer (computer scientist) | Erik Meijer]] and his team.
- Functional Reactive Programming has significantly influenced the design and philosophy of Reactive Extensions, with key influences including the origin of FRP, which was coined by [[Conal Elliott]] and Paul Hudak in their 1997 paper "Functional Reactive Animation", and the concept of describing time-varying values and event streams in a declarative way.
- The core ideas of original Functional Reactive Programming (FRP) include behaviors, which are continuous, time-parameterized values, and events, which are discrete, time-ordered occurrences, with a focus on pure denotational semantics, composition via higher-order functions, and no side effects or explicit state mutation.
- FRP's influence on [[ReactiveX | Reactive Extensions]] (Rx) is evident in several areas, including the concept of time-varying values, which is adapted in Rx as observables that emit multiple values over time, and event streams, which are directly influential in the creation of observable sequences of discrete events in Rx.
- The main lines of influence from FRP to Rx include the adaptation of declarative composition, which is achieved in Rx through the chaining of operators such as map, filter, merge, and switch, and the use of higher-order reactive values, which are similar to FRP's higher-order combinators and are implemented in Rx as flattening operators like flatMap, mergeMap, switchMap, and concatMap.
- Rx also preserves FRP's property of automatic propagation, where data flows through operator chains automatically, and includes time-based operators like delay, debounce, throttle, and sample, which echo FRP's focus on time-dependent behavior.
- However, there are key philosophical and practical divergences between FRP and Rx, including the fact that Rx is impure by design, embracing side effects and built for real-world imperative applications, whereas classic FRP is purely functional and denotational, and Rx unifies everything into discrete observable sequences, whereas FRP distinguishes between continuous behaviors and discrete events.
- The strength of FRP's influence on Rx varies across different aspects, with very strong influence in areas like declarative stream composition, operator vocabulary, and push-based event streams, but weaker influence in areas like pure functional semantics, continuous behaviors, and denotational precision.
- According to [[Erik Meijer (computer scientist) | Erik Meijer]], the principal designer of Rx, Reactive Extensions can be seen as taking the compositional spirit and event + stream ideas from FRP, but reformulating them for mainstream developers using a push-based, impure, discrete-only model that integrates well with existing imperative codebases.

## Influence of Functional Reactive Programming
- [[Reactive Streams]] is a standard specification for asynchronous stream processing with non-blocking backpressure, primarily targeting the JVM ecosystem but designed to be language-agnostic in its core principles, and its primary goal is to govern the exchange of stream data across asynchronous boundaries without overwhelming consumers.
- The Reactive Streams specification defines a minimal set of four interfaces, including Publisher, Subscriber, Subscription, and Processor, which form the entire contract and are very intentionally minimal, with the central innovation being mandatory non-blocking backpressure via the demand mechanism.
- The demand mechanism works by having a Subscriber call request(n) to signal how many elements it is ready to receive, and a Publisher must never emit more elements than the total outstanding demand, with the demand signal flowing upstream from consumer to producer, creating a dynamic, cooperative flow-control system.
- Reactive Streams differs from [[ReactiveX | Reactive Extensions]] (Rx) in several aspects, including backpressure, which is mandatory and non-blocking in Reactive Streams, whereas it is optional or library-specific in Rx, and the core contract, which includes Publisher, Subscriber, Subscription, and Processor in Reactive Streams, whereas Rx has Observable, Observer, and Subscription.
- The Reactive Streams initiative started around 2013-2014 to unify the reactive landscape on JVM, and modern JVM reactive libraries, including [[RxJava]] 2, Reactor, [[Akka (toolkit) | Akka]] Streams, and others, implement Reactive Streams and are TCK-verified, with the latest stable version being 1.0.4, which is very stable and has not required major changes since 1.0.0.
- Reactive Streams is not a full reactive programming library, but rather a standard for asynchronous stream processing with non-blocking backpressure, and it has become a [[Java (programming language) | Java]] standard library, with java.util.concurrent.Flow being a direct copy of the Reactive Streams interfaces, and its official site and reference implementation can be found at [https://www.reactive-streams.org/](https://www.reactive-streams.org/) and [https://github.com/reactive-streams/reactive-streams-jvm](https://github.com/reactive-streams/reactive-streams-jvm), respectively.
- The [[Reactive Streams]] specification is a low-level interoperability protocol and backpressure contract that enables libraries such as Project Reactor, RxJava 2/3, Akka Streams, Vert.x, Spring WebFlux, and Kafka Reactive Streams connectors to safely exchange streams without causing memory issues or requiring libraries to understand each other's internal buffering strategies.
- The rich operators, including map, flatMap, filter, debounce, window, and others, are provided by the libraries that implement Reactive Streams, rather than the specification itself, and are used by developers on a daily basis to work with streams.
- RxJS can be understood as a combination of the Iterator Pattern and the Subject/Observer Pattern, which provides a deliberate and elegant unification of two classic design patterns, and is one of the most important conceptual foundations of Reactive Extensions.

## Comparison with Other Reactive Programming Concepts
- The Iterator Pattern represents pull-based sequential access to a collection of values, where the consumer pulls values from the producer, and is characterized by features such as synchronous access by default, lazy value production, and explicit completion via a done signal.
- The Observer/Subject Pattern, on the other hand, is the push-based counterpart, where the producer decides when and how many values to send to the consumer, and is characterized by features such as asynchronous access, multiple consumers, and a lifecycle that includes onNext, onError, and onComplete signals.
- The combination of the Iterator Pattern and the Observer/Subject Pattern in RxJS allows for a powerful and flexible way of working with streams, and is a key concept in understanding how RxJS works and how it can be used to build reactive applications.

## The (Time, Value) Model for Observables
- RxJS is a library that unifies the Iterator and Observer patterns under a single abstraction, allowing for a symmetric treatment of pull-based and push-based sequences, with Observables being the push-based counterpart to Iterables.
- The key design decision in RxJS is that an Observable is to push sequences what an Iterable is to pull sequences, enabling a unified approach to handling both synchronous and asynchronous data streams.
- The main differences between Iterables and Observables lie in the time dimension, asynchrony, and concurrency, as well as cancellation and multiple subscribers, with Observables being inherently temporal and supporting multicast behavior.
- RxJS operators can be divided into two broad categories: value-based operators, which focus on transforming, filtering, and combining emitted values without regard to timing, and time-based operators, which primarily care about when values arrive, how long something takes, or the temporal relationship between events.
- Value-based operators include transformation operators like map and scan, filtering operators like filter and distinctUntilChanged, inspection operators like tap, aggregation operators like reduce, conditional operators like find and every, and combining operators like withLatestFrom and combineLatest.
- Time-based operators, on the other hand, are concerned with the timing and temporal aspects of the data stream, and are used to answer questions like "when" or "how long" something happens, with examples including delay, debounce, and throttle, although these specific operators are not mentioned in the provided text.
- The unification of Iterables and Observables in RxJS allows developers to use a consistent mental model and operator vocabulary for both finite synchronous collections and infinite or asynchronous event streams, making the library feel powerful and flexible once this symmetry is understood.
- The library's design enables a wide range of use cases, from handling synchronous data streams to dealing with complex asynchronous event streams, and provides a rich set of operators for transforming, filtering, and combining data, making it a versatile tool for reactive programming.
- The RxJS operators can be categorized into several groups, including throttling/debouncing, delaying, time windows/buffering, sampling, timeout/race, taking/skipping by time, repeating/polling, and scheduling/observation, each with its own set of examples and core purposes.
- These operators introduce or manipulate time windows, delays, throttling, or debouncing, and are used to control emission rates, shift streams in time, group values, emit latest values at regular intervals, react to time-based failure conditions, and stop or skip based on timing events.

## Types of Operators and Their Behaviors
- The typical question that these operators answer is "When should this value be emitted/suppressed/grouped/reacted to?", and they are often used in combination with value-based operators to create complex pipelines.
- Some operators, such as take, first, and last, can sit in a gray area and behave as either time-based or value-based depending on usage, while higher-order operators like mergeMap, switchMap, and concatMap are usually value-based but influence timing behavior through inner subscriptions.
- The operators can be classified into two main focus areas: value-based, which concerns what the data is, and time-based, which concerns when the data arrives, with representative operators including map, filter, and debounceTime, and most common use cases including data transformation, selection, rate limiting, grouping, and timing control.
- RxJS Observables can be conceptualized as a lazy, potentially infinite sequence of pairs of time and value, which provides a precise and mathematically elegant way to understand the underlying mechanics of RxJS.
- The concept of an Observable in RxJS can be understood as a lazy, potentially infinite sequence of timestamped values, denoted as Observable<A> ≈ () → Sequence<(T, a)>, where Time is monotonically increasing and T ∈ ℝ⁺ ∪ {∞}, a ∈ A, providing a powerful mental model for understanding RxJS.
- This model is characterized by several key properties, including laziness, where nothing happens until subscription, push-based, where the producer decides when values arrive, potentially infinite, where the sequence may continue indefinitely, monotonic time, where values arrive in order over time, discrete events, where each pair (tᵢ, aᵢ) is a discrete occurrence, termination, where the sequence ends or errors, and asynchrony, where values can arrive later, on different threads.
- The (Time, Value) model can be visualized as a sequence of timestamped pairs, such as (42ms, 1), (180ms, 4), and (1200ms, "done"), with the sequence ending with a complete() or error() signal, and this view helps explain common operators, including map(f), filter(p), delay(d), debounceTime(ms), throttleTime(ms), bufferTime(n), sampleTime(n), auditTime(n), and switchMap(f), which become more intuitive when seen through this lens.
- Many RxJS operators can be understood in terms of their effect on the (Time, Value) model, such as map(f) applying a function to every value while keeping original timestamps, filter(p) keeping only pairs where p(a) = true while preserving original times, and delay(d) shifting every timestamp by +d, demonstrating the power and flexibility of this mental model.
- The (Time, Value) model also provides a useful comparison with other abstractions, including Array / Iterable, Promise, RxJS Observable, [[Reactive Streams]] Publisher, and Classic FRP Behavior, highlighting the unique characteristics of RxJS Observables, such as implicit time, laziness, potential infinity, and push-based behavior, which distinguish them from other programming constructs.
- Overall, the view of an Observable as a lazy, potentially infinite sequence of timestamped values provides a clean and powerful mental model for understanding RxJS, making time-dependent operators particularly intuitive and explaining why RxJS is fundamentally different from both arrays and promises, and this model can be used to explore how particular operators transform the sequence of timestamped pairs, as well as its relationship to hot vs cold observables, multicasting, or schedulers.
- The RxJS Observable can be viewed as a lazy sequence of timestamped pairs, denoted as `source$ = [ (t₁, v₁), (t₂, v₂), (t₃, v₃), … ]`, where `t₁ ≤ t₂ ≤ t₃ ≤ …` are monotonically non-decreasing timestamps and `vᵢ` is the emitted value, which helps in understanding how operators behave.
- There are several types of operators that transform this sequence, including pure value transformations, time-shifting operators, rate-limiting or suppressing operators, and windowing or grouping by time operators, each with its own set of rules and effects on the sequence.
- Pure value transformations, such as `map(f)`, `mapTo(constant)`, `pluck('x')`, `filter(p)`, `distinctUntilChanged()`, `distinct()`, and `tap(f)`, apply a function to the value but preserve the original timestamps, allowing for various operations like mapping, filtering, and distinct value handling.
- Time-shifting operators, including `delay(d)` and `delayWhen(fn)`, change the timestamps themselves by shifting events forward by a fixed duration or a duration determined by a function, while `observeOn(scheduler)` changes when notifications are delivered to the observer without altering emission times.
- Rate-limiting or suppressing operators, such as `debounceTime(ms)`, `throttleTime(ms, {leading: true})`, `throttleTime(ms, {leading: false})`, `auditTime(ms)`, and `sampleTime(ms)`, drop or postpone some pairs based on timing rules, allowing for debouncing, throttling, and sampling of the sequence.
- Windowing or grouping by time operators, including `bufferTime(n)`, `windowTime(n)`, and `bufferToggle(openings$, closingFn)`, collect values into groups based on time intervals, enabling buffering, windowing, and toggling of the sequence.
- The section discusses higher-order mapping operators in RxJS, which project each value into a new inner sequence and preserve the timing of inner emissions relative to when the inner observable was subscribed, with key operators including mergeMap, concatMap, switchMap, and exhaustMap.

## Domain-Agnostic Nature of RxJS Operators
- The text also highlights the flexibility and elegance of RxJS, where the domain can change, such as from web animations to GPS sensor data, but the operators remain the same, allowing for a separation between domain-specific producers and domain-agnostic combinators.
- The RxJS library provides a wide range of Observable creation functions that can be used to handle different domains and use cases, such as [[Document Object Model | DOM]] events, Web Animations, GPS/Geolocation, [[Server-sent events | Server-Sent Events]], [[WebSocket]] messages, HTTP polling, Mouse/Touch position, Audio/Video timeupdate, Game loop/physics simulation, and Financial ticks/market data.
- Regardless of the domain or data source, the same RxJS operators can be used to express business logic, allowing for reusable logic, abstraction over the data source, a consistent mental model for developers, and improved testability.
- The use of the same operators across different domains enables the creation of domain-independent reactive logic, which is a key feature of RxJS, and allows developers to reason about streams and transformations in the same way, regardless of the underlying domain.
- The fundamental RxJS equation is based on the separation of Observable creation strategies, value types, and arrival patterns from the operators and composition language used to transform and combine the data, resulting in a powerful and flexible library for handling rich, interactive, real-time, or multi-source applications.
- The RxJS library provides a range of combining operators, such as combineLatest, withLatestFrom, merge, zip, and raceWith, that can be used to combine small, focused streams from different domains, enabling the creation of complex and powerful reactive systems.
- The use of RxJS enables developers to write business logic that is decoupled from the underlying data source, allowing for greater flexibility and reuse of code, and making it easier to test and maintain complex reactive systems.
